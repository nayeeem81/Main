When your multi-tenant system dictates that an email address can only be used once across the entire application, you are effectively building a Single Identity Provider (IdP) system. Because the user identity is unique across the whole platform, your architecture must strictly decouple Authentication (who the user is) from Tenant Authorization (which tenant they are allowed to access).

Here is the best-practice policy and architectural strategy for this specific scenario.

1. The Global Identity Policy

Global Users Table: Users must be stored in a central database table that is completely outside the tenant-isolated data models. The Email column must have a global UNIQUE constraint.

Tenant Mapping Table: Create a many-to-many relationship table (e.g., UserTenants) that links a single UserId to multiple TenantId records, along with their tenant-specific roles.

public class UserTenant
{
    public string UserId { get; set; } // Points to global user
    public string TenantId { get; set; } // Points to tenant
    public string Role { get; set; } // Tenant-specific role (e.g., "Admin", "User")
}

2. Best Practice Workflow by Tenant Type
Since you allow access via Domain, Subdomain, and Subdirectory Profiles, here is how your email and routing policy should handle them.

Option A: Tenant by Domain or Subdomain (tenant1.com or ://app.com)

This is the trickiest pattern because the user navigates directly to a tenant space before logging in.

The Navigation: User goes to ://app.com.
The Verification: They enter their email. Your system checks the global user database.

The Validation:

If the email exists and is mapped to Company A, log them in. If the email exists but is NOT mapped to Company A, deny access to this subdomain, but inform them they have an account on the platform.

Security Note: To prevent user enumeration attacks, use generic messaging like: "If this email belongs to an authorized account for this workspace, an email has been sent."

Option B: Tenant Profile under Central Domain (://app.com)
This is the most natural workflow for a unique email policy. 

The Central Login: Users always log in at a single, root URL: ://app.com. 
The Tenant Resolver Redirect: Once authenticated, read their mapped tenants from the database:

If mapped to 1 tenant: 
Seamlessly redirect them straight to ://app.com.
If mapped to multiple tenants: Present a "Choose your Workspace" selection screen, then route them to their chosen profile.

Enforce Centralized Sign-Up: 

Users should register accounts via your central portal, not inside a specific tenant dashboard, to prevent accidental duplicate account attempts.

Account Merging Strategy: If an administrator from Tenant B invites an email that already exists under Tenant A, the system should not create a new user. It should simply insert a new record into the UserTenants mapping table and send an invitation link.

Sanitize Domain Routing in Middleware: Your existing TenantResolverMiddleware must be updated. Once a user is authenticated, it should cross-reference tenantContext.

3. Implementing Token Generation Code (.NET 8)

Here is how you generate a tenant-scoped JWT during login. It looks up the user globally but embeds the specific tenant mapping and role into the claims.

using System.IdentityModel.Tokens.Jwt;
using System.Security.Claims;
using Microsoft.IdentityModel.Tokens;

public class TokenService
{
    public string GenerateTenantToken(ApplicationUser user, UserTenant activeTenantMapping)
    {
        var claims = new List<Claim>
        {
            new Claim(ClaimTypes.NameIdentifier, user.Id),
            new Claim(ClaimTypes.Email, user.Email),
            
            // CRITICAL: Embed the context of the current active tenant
            new Claim("tenant_id", activeTenantMapping.TenantId),
            new Claim(ClaimTypes.Role, activeTenantMapping.TenantRole) 
        };

        var key = new SymmetricSecurityKey(Encoding.UTF8.GetBytes("YourSuperSecretKeyThatIsLongEnough123!"));
        var creds = new SigningCredentials(key, SecurityAlgorithms.HmacSha256);

        var token = new JwtSecurityToken(
            issuer: "your-identity-provider",
            audience: "your-application",
            claims: claims,
            expires: DateTime.UtcNow.AddHours(2),
            signingCredentials: creds
        );

        return new JwtSecurityTokenHandler().WriteToken(token);
    }
}

Would you like to explore how to design the "Switch Workspace" logic so a user can jump from Tenant A to Tenant B seamlessly using these tokens?

Yes, please

To let a user switch between workspaces (e.g., from Tenant A to Tenant B) seamlessly, your application must exchange their current token for a new one containing the target tenant's ID and role.

This process must securely verify that the global user actually owns a membership in the target tenant before granting access. Here is the step-by-step implementation for workspace switching.

1. Create a Workspace Switching Controller
This controller exposes an endpoint that takes a targetTenantId. It verifies the logged-in user's relationship with that tenant and issues a fresh token.

using Microsoft.AspNetCore.Authorization;
using Microsoft.AspNetCore.Mvc;
using System.Security.Claims;

[Authorize]
[ApiController]
[Route("api/auth")]
public class WorkspaceController : ControllerBase
{
    private readonly ApplicationDbContext _context;
    private readonly TokenService _tokenService;

    public WorkspaceController(ApplicationDbContext context, TokenService tokenService)
    {
        _context = context;
        _tokenService = tokenService;
    }

    [HttpPost("switch-workspace")]
    public async Task<IActionResult> SwitchWorkspace([FromBody] SwitchWorkspaceRequest request)
    {
        // 1. Extract the Global User ID from the current active token claims
        var userId = User.FindFirst(ClaimTypes.NameIdentifier)?.Value;
        if (string.IsNullOrEmpty(userId)) return Unauthorized();

        // 2. Fetch the user and verify they belong to the requested target tenant
        var tenantMapping = await _context.UserTenants
            .Include(ut => ut.User)
            .FirstOrDefaultAsync(ut => ut.UserId == userId && ut.TenantId == request.TargetTenantId);

        if (tenantMapping == null)
        {
            // Security: Prevent the user from accessing a tenant they don't belong to
            return Forbid("You do not have access to this workspace.");
        }

        // 3. Generate a brand new token containing the new tenant context
        var newToken = _tokenService.GenerateTenantToken(tenantMapping.User, tenantMapping);

        // 4. Fetch details about the target tenant to know where to redirect them
        var targetTenant = await _context.Tenants.FindAsync(request.TargetTenantId);

        return Ok(new SwitchWorkspaceResponse
        {
            Token = newToken,
            // Return target URL (e.g., "://app.com" or "/tenantb") so frontend knows where to route
            RedirectUrl = $"https://{targetTenant.Domain}/dashboard" 
        });
    }
}

public class SwitchWorkspaceRequest { public string TargetTenantId { get; set; } = string.Empty; }
public class SwitchWorkspaceResponse { public string Token { get; set; } = string.Empty; public string RedirectUrl { get; set; } = string.Empty; }

2. Frontend Lifecycle & Redirect Routing Logic

Depending on how you separate tenants visually, your application UI handles the response from the switch endpoint differently:

1. For Subdirectory Profiles (://app.com ➔ ://app.com)
The Flow: The UI requests the token change, stores the new JWT in local storage or cookies, and updates the URL path to /tenantB/dashboard.

Benefit: Very fast because the entire application context stays in the same browser tab without losing system state.

2. For Custom Domains / Subdomains (tenantA.com ➔ tenantB.com)
Browsers prevent sharing LocalStorage or cookies across different root domains (CORS security policies).

The Flow:
The user clicks "Switch to Tenant B". 

The current page calls the endpoint and receives the new token.

The current page appends the new token as a temporary query parameter and redirects the window browser: window.location.href = "" + response.token.

The landing page at tenantB.com extracts the token from the URL parameter, saves it to its own domain-specific storage, clears the URL query parameter, and logs the user in.

2. Implement the Authorization Handler
The handler acts as the security guard. It extracts the tenant_id and UserId directly from the user's incoming JWT token claims, checks the database via dependency injection, and validates their tenant-specific role.

using Microsoft.AspNetCore.Authorization;
using System.Security.Claims;
using Microsoft.EntityFrameworkCore;

public class TenantRoleHandler : AuthorizationHandler<TenantRoleRequirement>
{
    private readonly ApplicationDbContext _dbContext;

    // You can safely inject your scoped database context here
    public TenantRoleHandler(ApplicationDbContext dbContext)
    {
        _dbContext = dbContext;
    }

    protected override async Task HandleRequirementAsync(
        AuthorizationHandlerContext context, 
        TenantRoleRequirement requirement)
    {
        // 1. Extract the Global User ID from claims
        var userId = context.User.FindFirst(ClaimTypes.NameIdentifier)?.Value;
        
        // 2. Extract the Active Tenant ID from claims (baked into the token on login/switch)
        var tenantId = context.User.FindFirst("tenant_id")?.Value;

        if (string.IsNullOrEmpty(userId) || string.IsNullOrEmpty(tenantId))
        {
            context.Fail(); // No valid tenant context found in the token
            return;
        }

        // 3. Query the database junction table to check the role for this exact tenant
        var userTenantMapping = await _dbContext.UserTenants
            .AsNoTracking()
            .FirstOrDefaultAsync(ut => ut.UserId == userId && ut.TenantId == tenantId);

        // 4. Validate if the user belongs to the tenant and matches the required role
        if (userTenantMapping != null && userTenantMapping.TenantRole == requirement.RequiredRole)
        {
            context.Succeed(requirement); // Access granted!
        }
        else
        {
            context.Fail(); // Unauthorized inside this specific tenant
        }
    }
}

Why this design excels

Blazing Fast Evaluation: If you use a high-performance application setup, you can update the TenantRoleHandler to read the role directly from a "role" claim inside the JWT token instead of hitting the database on every action.

Granular Control: Because it evaluates both the UserId and the current tenant_id simultaneously, a user who tries to forge a URL request to another workspace will be immediately blocked with a 403 Forbidden response.

To make your code cleaner and more readable, you can wrap the policy check inside a custom attribute. This allows you to write [RequireTenantRole("Admin")] directly on top of your controllers instead of the more verbose [Authorize(Policy = "TenantAdminOnly")].

In .NET, the cleanest way to do this is by creating a custom attribute that implements IAuthorizationRequirementData. This interface maps the attribute parameters directly to your authorization pipeline automatically.Here is how to implement the custom attribute.

2. Create the Custom Attribute ClassCreate a class named RequireTenantRoleAttribute. It inherits from Attribute and implements IAuthorizationRequirementData.

using Microsoft.AspNetCore.Authorization;

[AttributeUsage(AttributeTargets.Class | AttributeTargets.Method, AllowMultiple = true)]
public class RequireTenantRoleAttribute : Attribute, IAuthorizationRequirementData
{
    public string Role { get; set; }

    public RequireTenantRoleAttribute(string role)
    {
        Role = role;
    }

    // This method automatically feeds our custom requirement into the .NET Core authorization framework
    public IEnumerable<IAuthorizationRequirement> GetRequirements()
    {
        yield return new TenantRoleRequirement(Role);
    }
}

public static class DatabaseInitializer
{
    public static async Task SeedDataAsync(IServiceProvider serviceProvider)
    {
        using var scope = serviceProvider.CreateScope();
        var context = scope.ServiceProvider.GetRequiredService<ApplicationDbContext>();
        var userManager = scope.ServiceProvider.GetRequiredService<UserManager<ApplicationUser>>();
        var roleManager = scope.ServiceProvider.GetRequiredService();

        // 1. Automatically run pending database migrations (Optional but highly recommended)
        await context.Database.MigrateAsync();

        // =======================================================
        // 2. SEED GLOBAL ROLES (For overall system admins, not tenant roles)
        // =======================================================
        string globalAdminRole = "GlobalAdmin";
        if (!await roleManager.RoleExistsAsync(globalAdminRole))
        {
            await roleManager.CreateAsync(new IdentityRole(globalAdminRole));
        }

        // =======================================================
        // 3. SEED MASTER/DEFAULT TENANT
        // =======================================================
        string masterTenantId = "master-tenant";
        var masterTenant = await context.Tenants
            .IgnoreQueryFilters() // Bypass any active global filters during initialization
            .FirstOrDefaultAsync(t => t.Id == masterTenantId);

        if (masterTenant == null)
        {
            masterTenant = new TenantInfo
            {
                Id = masterTenantId,
                Name = "System Master Workspace",
                Domain = "localhost", // Matches your local development environment url
                Type = TenantType.Enterprise
            };
            context.Tenants.Add(masterTenant);
            await context.SaveChangesAsync();
        }

        // =======================================================
        // 4. SEED SUPER ADMINISTRATOR (Global User)
        // =======================================================
        string adminEmail = "admin@system.com";
        var adminUser = await userManager.FindByEmailAsync(adminEmail);

        if (adminUser == null)
        {
            adminUser = new ApplicationUser
            {
                UserName = adminEmail,
                Email = adminEmail,
                EmailConfirmed = true
            };

            // Create the user with a default strong password
            var result = await userManager.CreateAsync(adminUser, "SecureP@ss123!");
            
            if (result.Succeeded)
            {
                // Assign to global administration role
                await userManager.AddToRoleAsync(adminUser, globalAdminRole);

                // Map this user as the 'Admin' of the Master Tenant
                var userTenantMapping = new UserTenant
                {
                    UserId = adminUser.Id,
                    TenantId = masterTenantId,
                    TenantRole = "Admin" // Matches your custom [RequireTenantRole("Admin")] setup
                };
                context.UserTenants.Add(userTenantMapping);
                await context.SaveChangesAsync();
            }
        }
    }
}

Structural Best Practices Implemented Here.

IgnoreQueryFilters() Safety: Since your ApplicationDbContext will likely filter data globally by TenantId, the seeder bypasses this filter explicitly. Otherwise, the seeder would fail to detect if the records already exist.

Idempotency (Run-Once Guardrails): The logic uses conditional checks (if (adminUser == null)) everywhere. This allows the seeder to run securely every time the application restarts without duplicating data or throwing primary key exceptions.

Scoped Lifecycle Protection: Database contexts and user managers cannot be safely extracted from a root provider directly. Using serviceProvider.CreateScope() creates an isolated execution container that automatically disposes of database connections cleanly when seeding completes.Now that your seeding framework is operational, what component would you like to build out next?
