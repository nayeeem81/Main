Tenant Secrets vs Tenant IDs

A Tenant Secret is not the TenantId. 

It is a secret key (like a cryptographic signature key or a client password) used to sign and verify JWT tokens specifically for that tenant.

Tenant ID: A unique public identifier (e.g., tenant-abc-123 or a GUID).

Tenant Table: A database table that maps the TenantId to things like database connection strings or configurations.

Tenant Secret: A private string used by your authentication server to generate JWT tokens. 

If every tenant has their own secret, your API needs to look up that secret to validate the JWT.

public class TenantResolverMiddleware
{
    private readonly RequestDelegate _next;

    public TenantResolverMiddleware(RequestDelegate next)
    {
        _next = next;
    }

    public async Task InvokeAsync(HttpContext context)
    {
        // 1. Extract the TenantId from the request header
        if (!context.Request.Headers.TryGetValue("X-Tenant-ID", out var headerTenantId))
        {
            context.Response.StatusCode = StatusCodes.Status400BadRequest;
            await context.Response.WriteAsync("Missing Tenant ID header.");
            return;
        }

        // 2. Extract the TenantId stored inside the Server Session
        var sessionTenantId = context.Session.GetString("TenantId");

        if (string.IsNullOrEmpty(sessionTenantId))
        {
            context.Response.StatusCode = StatusCodes.Status401Unauthorized;
            await context.Response.WriteAsync("No active session found for this tenant. Please login.");
            return;
        }

        // 3. Cross-validate the Header value against the Session value
        if (headerTenantId != sessionTenantId)
        {
            context.Response.StatusCode = StatusCodes.Status403Forbidden;
            await context.Response.WriteAsync("Tenant ID mismatch. Request rejected.");
            return;
        }

        // 4. Validation passed! Store it in HttpContext for downstream usage
        context.Items["CurrentTenantId"] = sessionTenantId;

        await _next(context);
    }
}

Since you are storing your JWT inside a secure HttpOnly Cookie, your application is highly secure against Cross-Site Scripting (XSS). However, relying on cookies automatically exposes your backend to Cross-Site Request Forgery (CSRF) attacks, because browsers automatically attach cookies to requests originating from external malicious sites.

To completely secure your Multi-Tenant app, we will implement an Anti-Forgery (CSRF) Token pattern that works seamlessly with your Tenant ID tracking.

Step 1: The Secure CSRF Strategy

When logging in: 

The server generates a unique, random string (the CSRF token).

The Response: 
The server sends this token back to the browser in a standard, readable JavaScript Cookie (e.g., XSRF-TOKEN).

The Front-End (Angular/React/Vue): 
Your front-end app reads this cookie using JavaScript and automatically attaches its value to a custom HTTP Header (e.g., X-XSRF-TOKEN) on every subsequent API call (POST, PUT, DELETE).

The Validation: The server compares the token sent in the custom header against the secure token it knows about. 

A malicious site can't steal the readable cookie due to browser security boundaries (SameSite=Strict), blocking the attack.

        {
            HttpOnly = true,
            Secure = true,
            SameSite = SameSiteMode.Strict,
            Expires = DateTime.UtcNow.AddHours(2)
        };
        Response.Cookies.Append("X-Auth-Token", tokenString, jwtCookieOptions);

        // 3. Set TenantId in server Session
        HttpContext.Session.SetString("TenantId", resolvedTenantId);

* Idempotency and Safety: Neither POST nor PATCH is strictly idempotent by definition, but POST is universally accepted for any state-changing operations that do not map perfectly to RESTful CRUD resources. [7, 8] 
* Browser and Framework Support: POST is completely seamless across all older browsers, firewalls, and proxy servers. Some legacy infrastructure can still occasionally strip out or block PATCH requests.

## When to Use Each Verb

* POST: Use this when you are submitting form data, triggering a process, logging a user in, or saving temporary wizard/multi-step data to the server session. [9, 10, 11] 
* PATCH: Use this exclusively when you have a specific database entity already saved, and you want to send a partial update (like changing just the phone number of User #12). [12]

Yes, you can dynamically configure antiforgery settings per tenant in ASP.NET Core 8.0, but you must use the Options Pattern correctly.
Changing the header name has absolutely no impact on whether the browser automatically sends the cookie to the server. Browsers send cookies automatically based only on the domain, path, and security flags, completely ignoring custom HTTP headers. [1, 2] 
However, you must change both the cookie name and the header name dynamically if your tenants expect different configurations, and you must use a custom filter or middleware to read them.
Here is exactly how this works and how to implement it.
------------------------------
## 1. Does changing the Header Name affect Cookie Delivery?

* No. Browsers automatically attach cookies to matching requests. They do not look at HTTP headers to decide this.
* Why the Header Name matters: In AJAX/Fetch requests (like SPAs), JavaScript manually reads a token from the page and places it into a custom header (e.g., X-XSRF-TOKEN). The server looks at that specific header name to find the token and validate it against the cookie. If your client-side JavaScript sends a different header name than what the server expects, validation will fail. [3, 4, 5, 6] 

------------------------------
## 2. How to Dynamically Set Cookie and Header per Tenant
Because AntiforgeryOptions is a singleton by default, you cannot just set it once at startup. You must use IOptionsSnapshot<AntiforgeryOptions> or a custom options setup that runs on every request.
Add this setup in your Program.cs: [7] 

using Microsoft.AspNetCore.Antiforgery;using Microsoft.Extensions.Options;

builder.Services.AddHttpContextAccessor();
// Dynamically configure Antiforgery per request/tenant
builder.Services.AddScoped<IConfigureOptions<AntiforgeryOptions>>, TenantAntiforgeryOptions>();
builder.Services.AddAntiforgery();

Create the dynamic configuration class:

public class TenantAntiforgeryOptions : IConfigureOptions<AntiforgeryOptions>
{
    private readonly IHttpContextAccessor _httpContextAccessor;

    public TenantAntiforgeryOptions(IHttpContextAccessor httpContextAccessor)
    {
        _httpContextAccessor = httpContextAccessor;
    }

    public void Configure(AntiforgeryOptions options)
    {
        var context = _httpContextAccessor.HttpContext;
        if (context == null) return;

        // Resolve your tenant identifier here (from host, route, header, etc.)
        string tenantId = context.Items["TenantId"]?.ToString() ?? "default"; 

        // Dynamically assign names based on tenant
        options.Cookie.Name = $".AspNetCore.Antiforgery.{tenantId}";
        options.HeaderName = $"X-XSRF-TOKEN-{tenantId.ToUpper()}"; 
    }
}

------------------------------
## 3. Do you need a custom Action Filter for validation?
No, you do not need a custom validation filter just because the cookie name changes.
If you use the IConfigureOptions approach shown above, ASP.NET Core's built-in [ValidateAntiForgeryToken] or [AutoValidateAntiforgeryToken] filters will automatically spin up on the request, fetch the scoped AntiforgeryOptions for that specific tenant, and look for the correct cookie name and header name automatically.

* Header Name: Only needs to change if your tenant frontend applications use different header names for JavaScript APIs.

## Critical Code Correction for ASP.NET Core 8.0

While your logic is sound, you must use IConfigureNamedOptions<AntiforgeryOptions> instead of the standard IConfigureOptions.

In ASP.NET Core, the Antiforgery system initializes its options using a specific internal name. If you use a standard IConfigureOptions, your dynamic changes might be skipped or overridden by the framework's internal defaults.

Update your TenantAntiforgeryOptions.cs to match this pattern:

using Microsoft.AspNetCore.Antiforgery;
using Microsoft.Extensions.Options;
public class TenantAntiforgeryOptions : IConfigureNamedOptions<AntiforgeryOptions>
{
    private readonly IHttpContextAccessor _httpContextAccessor;

    public TenantAntiforgeryOptions(IHttpContextAccessor httpContextAccessor)
    {
        _httpContextAccessor = httpContextAccessor;
    }

    // This method handles the named instances (Crucial for Antiforgery)
    public void Configure(string? name, AntiforgeryOptions options)
    {
        Configure(options);
    }

    public void Configure(AntiforgeryOptions options)
    {
        var context = _httpContextAccessor.HttpContext;
        if (context == null) return;

        // Resolve your tenant identifier (ensure this middleware runs AFTER Tenant resolution)
        string tenantId = context.Items["TenantId"]?.ToString() ?? "default"; 

        // 1. Keep the Header static across all tenants
        options.HeaderName = "X-XSRF-TOKEN";

        // 2. Isolate cookies per tenant
        options.Cookie.Name = $".AspNetCore.Antiforgery.{tenantId}";

        // 3. Isolate hidden form fields per tenant (for standard MVC/Razor forms)
        options.FormFieldName = $"__RequestVerificationToken_{tenantId}";
    }
}

You have hit on one of the most confusing parts of web security! Let’s break down exactly why AJAX/Fetch behaves differently than HTML forms, why the browser alone is not enough, and how you can use hidden input fields instead of reading cookies.
------------------------------
## 1. Why can't we just rely on the browser to send the cookie?
You are 100% correct that the browser automatically sends the cookie with your fetch or jQuery AJAX request (provided they are on the same domain).
The problem is that this automatic behavior is exactly what causes the CSRF vulnerability in the first place.

* The CSRF Attack Scenario: Imagine a user is logged into your application (tenant1.com). They accidentally visit a malicious website in another tab. That malicious site triggers a hidden AJAX request to ://tenant1.com.
* Because the request is going to tenant1.com, the browser automatically attaches your authentication and antiforgery cookies.
* If the server only checked the automatically sent cookie, it would think the request was legitimate and execute the deletion! [1, 2] 

The Defense: To prevent this, the server says: "I will not trust the cookie alone, because the browser sends it automatically without the user's explicit consent. The client must prove they are actually interacting with my real user interface by sending the token a second time via a custom HTTP header or a form body parameter."
Since a malicious third-party site cannot read cookies from your domain (due to the Same-Origin Policy), they cannot copy the token into a custom header. Only your legitimate JavaScript code can. [3] 
------------------------------
## 2. Can I read the hidden input field instead of the cookie?
Yes, absolutely! This is an excellent alternative that allows you to keep all your cookies HttpOnly = true (maximum security).
If your backend renders the page (e.g., an MVC view or Razor Page), the ASP.NET Core Form Tag Helper automatically injects the hidden field into the DOM. Your JavaScript can grab that value directly from the page and append it to your AJAX header or payload.
## How to implement this with jQuery / Fetch:
Step 1: Make sure the token is rendered on the page.
If you have a form on the page, the field is already there. If you don't have a form, you can manually inject it anywhere in your Layout or View HTML using the HTML helper:

@Html.AntiForgeryToken() 

Step 2: Read it via JavaScript and send it via the Static Header:
Using jQuery AJAX:

    success: function(response) { console.log(response); }

## 3. What about credentials: 'include' or allowCredentials?
Setting credentials: 'include' in Fetch (or withCredentials: true in jQuery/Axios) only ensures that the browser sends the cookies. [4] 

* If your frontend and backend are on the exact same domain/port (Monolith), credentials are sent automatically by default. You do not need to turn this on.
* If your frontend is on a different domain (Cross-Origin, like ://tenant1.com talking to ://tenant1.com), you must enable credentials so the browser attaches the session/antiforgery cookies. [5] 

However, enabling credentials does not bypass the need for the token verification. Even with credentials turned on, your server's antiforgery filter will still block the request with a 400 Bad Request unless your JavaScript explicitly reads the token (either from the hidden input field or a readable cookie) and sends it back in the X-XSRF-TOKEN header.

## 1. Global Setup for jQuery AJAX
Add this script to your global JavaScript file or layout. It intercepts every jQuery AJAX request before it is sent to the server.

$(document).ajaxSend(function (event, xhr, settings) {
    // Only apply to state-changing requests (ignore GET, HEAD, OPTIONS)
    if (!/^(GET|HEAD|OPTIONS|TRACE)$/i.test(settings.type)) {
        
        const tenantId = window.CurrentTenantId || "default";
        
        // Find the dynamic input field using the tenant ID
        const tokenInput = $(`input[name="__RequestVerificationToken_${tenantId}"]`);
        
        if (tokenInput.length > 0) {
            // Automatically inject the static header name with the token value
            xhr.setRequestHeader("X-XSRF-TOKEN", tokenInput.val());
        }
    }
});


* How it works: Whenever you run $.post(), $.ajax(), or $.delete(), jQuery fires the ajaxSend event globally. This script intercepts it, validates the HTTP method, finds your tenant field, and injects the header seamlessly.

------------------------------
## 2. Global Setup for Fetch API
Because the native fetch() API does not have built-in global configuration hooks like jQuery, we must create a clean global wrapper.
Add this script to your layout before any of your application scripts run:

(function () {
    const originalFetch = window.fetch;

    window.fetch = async function (resource, config) {
        // Initialize config object if it doesn't exist
        config = config || {};
        const method = config.method ? config.method.toUpperCase() : "GET";

        // Only modify state-changing requests
        if (!/^(GET|HEAD|OPTIONS|TRACE)$/i.test(method)) {
            
            const tenantId = window.CurrentTenantId || "default";
            const tokenInput = document.querySelector(`input[name="__RequestVerificationToken_${tenantId}"]`);

            if (tokenInput && tokenInput.value) {
                // Initialize headers object correctly
                config.headers = config.headers || {};

                if (config.headers instanceof Headers) {
                    config.headers.set("X-XSRF-TOKEN", tokenInput.value);
                } else if (Array.isArray(config.headers)) {
                    config.headers.push(["X-XSRF-TOKEN", tokenInput.value]);
                } else {
                    config.headers["X-XSRF-TOKEN"] = tokenInput.value;
                }
            }
        }

        // Execute the actual native fetch call with our modified config
        return originalFetch(resource, config);
    };
})();


* How it works: This is a classic JavaScript technique known as "monkey-patching." It safely intercepts the native browser fetch function, injects your custom header logic on non-GET requests, and then forwards the call to the browser's real engine.

## 2. When is it executed?
An Action Filter executes inside the MVC Pipeline, which happens much later than the standard HTTP middleware pipeline. [4] 

The order of execution for a request looks like this:

   1. HTTP Request enters the app.
   2. Standard Middleware runs (e.g., UseRouting, UseAuthentication, your tenant resolution middleware).
   3. Endpoint Routing matches the request to a specific Controller Action.
   4. AntiforgeryActionFilter executes (Specifically, its OnActionExecuting method runs before your controller code starts).
   5. Your Controller Action code runs. [5, 6, 7, 8] 

------------------------------
## 3. Does it execute for basic form submissions, or only AJAX?

It will execute for BOTH.

Action filters do not care how the browser sent the request. Whether it is a traditional HTML form submission, a jQuery $.ajax call, or a fetch() request, if it hits a controller action covered by this filter, the filter will execute. [9, 10] 

Inside the filter's code, ASP.NET Core's IAntiforgery engine is smart enough to handle both types of payloads seamlessly:

* For basic forms: It looks for the hidden form field matching your dynamic FormFieldName (e.g., __RequestVerificationToken_tenant1).
* For AJAX/Fetch: It looks for the custom HTTP header matching your HeaderName (e.g., X-XSRF-TOKEN). [11]

If you set the tenant inside an Action Filter, it happens too late. The antiforgery system will have already tried to spin up using a "default" or empty tenant context.

* The Fix: Tenant resolution (figuring out the tenant from the URL, header, or host) should always happen early in a Custom Middleware (app.UseMiddleware<TenantMiddleware>()). Your AntiforgeryActionFilter should only be responsible for validating the token once the tenant is already known.

## 3. The Dangerous Hidden Trap ⚠️

There is a specific reason why you might be using a custom AntiforgeryActionFilter instead of ASP.NET Core's built-in [ValidateAntiForgeryToken] attribute.

If you use the built-in [ValidateAntiForgeryToken] attribute, it acts as an Authorization Filter.

using Microsoft.AspNetCore.Antiforgery;
using Microsoft.AspNetCore.Mvc;
using Microsoft.AspNetCore.Mvc.Filters;

public class AntiforgeryActionFilter : IAsyncActionFilter
{
    private readonly IAntiforgery _antiforgery;
    private readonly ITenantSetter _tenantSetter; // Your tenant container

    public AntiforgeryActionFilter(IAntiforgery antiforgery, ITenantSetter tenantSetter)
    {
        _antiforgery = antiforgery;
        _tenantSetter = tenantSetter;
    }

    public async Task OnActionExecutionAsync(ActionExecutingContext context, ActionExecutionDelegate next)
    {
        // 1. Skip safe HTTP methods (GET, HEAD, OPTIONS)
        var method = context.HttpContext.Request.Method;
        if (HttpMethods.IsGet(method) || HttpMethods.IsHead(method) || HttpMethods.IsOptions(method))
        {
            await next();
            return;
        }

        try
        {
            // 2. This triggers the validation using the dynamic options 
            //    that read from your scoped ITenantSetter
            await _antiforgery.ValidateRequestAsync(context.HttpContext);
        }
        catch (AntiforgeryValidationException)
        {
            // 3. Return a clean 400 Bad Request if validation fails
            context.Result = new BadRequestResult();
            return;
        }

        // 4. Continue to the controller action if validation passes
        await next();
    }
}

In ASP.NET Core, an Action Filter works like a pipeline wrapper around your controller method. Think of it like a security guard standing at a doorway.
When a request arrives, the filter code runs before your controller code.
Calling await next(); is how the filter tells ASP.NET Core: "This request is cleared. Let it pass through to the controller method."
Here is exactly what happens if you handle it incorrectly:
------------------------------
## Scenario 1: What happens with await next(); (Correct)
When a user requests a safe page via a GET request (e.g., viewing a product page), your filter checks the HTTP method, sees it is a GET, and hits await next();.
The framework immediately routes the request to your controller action, the page renders, and the user sees the website. Everything works perfectly.
------------------------------
## Scenario 2: What happens if you forget await next();?
If you simply typed return; without calling await next();, you would break your entire application.

// ❌ DO NOT DO THISif (HttpMethods.IsGet(method))
{
    return; // Short-circuits the pipeline!
}

In an ASP.NET Core filter, returning early without executing next() tells the framework to short-circuit and stop immediately. It assumes the filter has finished the request. Because no data or HTML view was ever sent by the controller, the user will just get a completely blank white screen with a 200 OK status code. Your controller actions for GET requests will never execute.
------------------------------
## Scenario 3: What happens if you don't skip GET requests at all?
If you removed the if block entirely and forced the filter to run validation on every single request:

_antiforgery.ValidateRequestAsync(context.HttpContext);

Your POST and DELETE requests would work, but every single GET request would fail with a 400 Bad Request.

When a user typed your URL into their browser, clicked a bookmark, or refreshed the page, the browser would send a standard GET request. Because standard browser navigation cannot carry an antiforgery token in a custom header or form field, the validation would fail, completely locking users out of your site.

You should use await next(); return; in both places, but for entirely different reasons.
To keep your code clean and prevent bugs, you should use an early return for safe methods (GET) to exit the filter immediately, and a final return at the very bottom of the method for unsafe methods (POST/DELETE) after validation passes.

Here is exactly how the logic must look inside your global filter:

public async Task OnActionExecutionAsync(ActionExecutingContext context, ActionExecutionDelegate next)
{
    var method = context.HttpContext.Request.Method;

    // 1. FOR SAFE METHODS (GET, HEAD, OPTIONS):
    // We execute the controller immediately and exit the filter early.
    if (HttpMethods.IsGet(method) || HttpMethods.IsHead(method) || HttpMethods.IsOptions(method))
    {
        await next(); 
        return; // MANDATORY: Stops this filter from running the validation code below!
    }

    // 2. FOR UNSAFE METHODS (POST, PUT, DELETE):
    // Run your tenant-specific antiforgery validation first.
    try
    {
        await _antiforgery.ValidateRequestAsync(context.HttpContext);
    }
    catch (AntiforgeryValidationException)
    {
        context.Result = new BadRequestResult(); // Short-circuit with a 400 error
        return; // Stops execution; the controller action is NEVER executed.
    }

    // 3. IF VALIDATION PASSES:
    // Execute the controller action.
    await next();
    return; // Optional but good practice to explicitly close the method.
}

------------------------------
## Why the return; inside the first if statement is mandatory:

If you remove the return; inside the if statement for safe methods, the code will execute await next() (running your controller and loading the page), but then it will keep reading down the file into the try/catch block.

It will then attempt to run _antiforgery.ValidateRequestAsync() on your GET request. Because GET requests do not have tokens, it will throw an exception, execute the catch block, overwrite your successful page result with a BadRequestResult(), and crash your user's browser navigation.
## Why you use the _ = await next(); syntax sometimes:

You might see _ = await next(); in some advanced tutorials. The underscore (_) is a discard variable. It is only used if you want to capture the final result context of the executed controller action but you don't plan on inspecting it.
For your antiforgery filter, you do not need to capture the result. Writing a clean, simple await next(); is perfectly sufficient, highly readable, and matches the standard ASP.NET Core architectural patterns.
