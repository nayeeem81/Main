Using [FluentValidation](https://github.com/FluentValidation/FluentValidation) for domain validation is a highly debated topic in software engineering. While it provides an incredibly clean, expressive, and strongly-typed API for data checking, applying it directly inside your Domain Layer often violates core Domain-Driven Design (DDD) principles. [1, 2, 3] 
Here is a breakdown of why this architectural tension exists, the best-practice approaches, and how to implement it correctly.
------------------------------
## The Architectural Conflict
In Clean Architecture and DDD, your core Domain Layer should remain pure. [1] 

* 
* Input Validation vs. Domain Invariants: Input validation (e.g., checking if an email is formatted correctly on a DTO) belongs in the API or Application layer. Domain validation enforces invariants—business rules that must always remain true for an entity (e.g., "An order cannot be approved if the customer is deactivated"). [1, 4] 
* Third-Party Coupling: Your domain models should ideally be plain objects (POCOs) free from infrastructure or external library dependencies. Coupling your domain directly to FluentValidation.AbstractValidator<T> breaks this isolation. [1, 3] 
* Always-Valid State: Domain entities should guard themselves dynamically. An entity shouldn't be allowed to exist in an invalid state. FluentValidation, by design, requires you to instantiate an object first and validate it afterwards. [1, 3, 5, 6] 
* 

------------------------------
## Recommended Patterns for Domain Validation## Approach 1: Guard Clauses in Entities (Standard DDD) [1, 7, 8] 
Instead of a framework, use constructor parameters and Guard Clauses directly inside your domain objects to enforce an "Always-Valid" state. Throw exceptions if business rules are broken. [1, 5, 7] 

public class Customer
{
    public Guid Id { get; private set; }
    public string Name { get; private set; }

    public Customer(Guid id, string name)
    {
        // Enforce invariants immediately
        if (id == Guid.Empty) throw new ArgumentException("Invalid ID");
        if (string.IsNullOrWhiteSpace(name)) throw new ArgumentException("Name cannot be empty");

        Id = id;
        Name = name;
    }
}

## Approach 2: FluentValidation in the Application Layer (Recommended) [9, 10] 
Keep FluentValidation out of your entities. Instead, use it on incoming DTOs, Commands, or Requests in your Application Layer to act as an anti-corruption layer before your domain layer is even touched. [1, 4, 11, 12, 13] 

// The Request DTO (Application Layer)public record CreateCustomerCommand(string Name, string Email);
// The Validator (Application Layer)public class CreateCustomerCommandValidator : AbstractValidator<CreateCustomerCommand>
{
    public CreateCustomerCommandValidator()
    {
        RuleFor(x => x.Name).NotEmpty().MaximumLength(100);
        RuleFor(x => x.Email).NotEmpty().EmailAddress();
    }
}

## Approach 3: Separated Domain Validators (Hybrid Approach) [6, 14] 
If your domain logic involves highly complex, multi-property business rules that change frequently, you can use FluentValidation. However, isolate the validator classes in a separate project or an application adapter layer so they don't pollute the core entities. [1, 15] 

// Core Domain (Pure)public class Order
{
    public List<OrderItem> Items { get; set; } = new();
    public decimal TotalAmount { get; set; }
    public bool IsExpressShipping { get; set; }
}
// Validation Layer (Interacts with the Domain)public class OrderBusinessRuleValidator : AbstractValidator<Order>
{
    public OrderBusinessRuleValidator()
    {
        // Complex rule: Express shipping requires a minimum spend
        RuleFor(order => order.TotalAmount)
            .GreaterThanOrEqualTo(50)
            .When(order => order.IsExpressShipping)
            .WithMessage("Express shipping requires a minimum order total of $50.");
    }
}

------------------------------
## External Logic & Asynchronous Domain Checks
One major benefit of FluentValidation over Data Annotations is the ability to easily perform asynchronous operations (like querying a database via a repository to check for uniqueness) using .MustAsync(). [3, 16] 
If you use this for domain uniqueness rules, inject your interfaces right into the validator constructor: [3, 16] 

public class UniqueEmailValidator : AbstractValidator<Customer>
{
    private readonly ICustomerRepository _repository;

    public UniqueEmailValidator(ICustomerRepository repository)
    {
        _repository = repository;

        RuleFor(customer => customer.Email)
            .MustAsync(async (email, cancellation) => 
                !await _repository.ExistsByEmailAsync(email, cancellation))
            .WithMessage("This email address is already in use.");
    }
}
