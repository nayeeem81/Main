Injecting a TenantId into log context isolates customer logs and enables end-to-end tracing across microservices. 

It is foundational for multi-tenant SaaS compliance, allowing you to troubleshoot customer-specific issues or filter activity in centralized observability platforms without sifting through noise.

1. Where to Do This (Injecting TenantId)

Do not pass TenantId manually as a parameter to every logging call. Instead, use language-specific contextual loggers (like Java’s MDC, C#'s ILogger.BeginScope, or Node.js's AsyncLocalStorage).

API Gateway: Intercept the incoming request at the API Gateway. Extract the TenantId from the JWT/Header, validate it, and inject it as a standard HTTP header (e.g., X-Tenant-ID) for downstream services.

Authentication/Tenant Middleware: 
In every downstream microservice, create a middleware layer. This middleware intercepts the X-Tenant-ID header, validates it, and binds it to the request’s execution context.

Log Enrichment: 
Configure your structured logging framework (e.g., Serilog, Winston, or Logback) to automatically read this execution context and append the TenantId attribute to every JSON log payload globally.

2. What to Log & Row-Level Tracing

Instead of dumping full database tables into logs, you should trace the operations affecting those tables.

What to Log:

Request lifecycle events:
1. (Start, End, Latency), CRUD operations on database records (e.g., "User updated row 45 in Customers table").

2. Security events: Authentication, failed permission checks, System faults and external API failures.

3. Logging Rows/Collections: 
- Never log the entire dataset (e.g., a "dump" of every row in a tenant's table). It creates massive log bloat and violates PII/GDPR compliance. 
- Instead, log the query parameters (e.g., { "tenantId": "T-99", "table": "invoices", "action": "SELECT", "limit": 50, "offset": 0 }). 
- For inserts or updates, log the affected record's primary keys or a cryptographic hash of the changed row.3. 

## Do You Need a Separate Log Database? 
Yes, but not necessarily a separate database engine.

Do not mix application logs with application data in the same database instance. While you can use the same logging cluster (like OpenSearch or Elasticsearch) for all tenants, you need to enforce 
1. Log Segregation & 
2. Role-Based Access Control (RBAC).
3. Shared Centralized Storage

## Shared Centralized Storage:
Send all microservice logs to a unified observability stack (e.g., Datadog, Elasticsearch, or Loki). This allows you to reconstruct a multi-service trace spanning 5 different microservices for a single tenant.

## Access Control: Do not give tenants direct access to the raw logs. Instead, use  Role-Based Access Control on your log indexers to ensure support engineers only see tenant-appropriate logs.4. 

# Other Features to Care About
When architecting multi-tenant logging, consider these critical constraints:

## PII & Data Masking: 
Never log unmasked customer secrets, passwords, or PII (Personally Identifiable Information).

## Correlation IDs (Trace IDs): 
Beyond TenantId, inject a unique TraceId per request. This ID must be passed to downstream microservice calls so you can track a specific multi-step user journey.

## Log Sampling/Rate Limiting: 
Aggressive users can flood your system with logs, incurring huge storage costs and system lag. Implement rate-limiting per tenant and log sampling during high-traffic periods.

## Async & Buffered Logging: 
Ensure your logging pipeline is asynchronous to prevent the logging thread from blocking critical business transactions.

public class ApplicationDbContext : DbContext
{
    private readonly string _currentTenantId;

    public ApplicationDbContext(DbContextOptions<ApplicationDbContext> options, ITenantService tenantService) : base(options)
    {
        _currentTenantId = tenantService.GetCurrentTenantId();
    }

    public override async Task<int> SaveChangesAsync(CancellationToken cancellationToken = default)
    {
        // Track modified/added entries before saving
        var auditEntries = OnBeforeSaveChanges();

        var result = await base.SaveChangesAsync(cancellationToken);

        // Save audit entries to an AuditLogs table or push to Serilog
        await SaveAuditLogsAsync(auditEntries); 

        return result;
    }

    private List<AuditEntry> OnBeforeSaveChanges()
    {
        ChangeTracker.DetectChanges();

        var auditEntries = new List<AuditEntry>();

        foreach (var entry in ChangeTracker.Entries())
        {
            if (entry.Entity is AuditLog || entry.State == EntityState.Detached || entry.State == EntityState.Unchanged)
                continue;

            var auditEntry = new AuditEntry
            {
                TenantId = _currentTenantId,
                TableName = entry.Metadata.GetTableName(),
                Action = entry.State.ToString(),
                PrimaryKey = entry.Properties.FirstOrDefault(p => p.Metadata.IsPrimaryKey())?.CurrentValue?.ToString()
            };

            // Capture old and new values for updates to trace data row changes

            auditEntries.add(auditEntry);

        }

        return auditEntries;
    }
}

LoggerMessageAttribute is a built-in .NET feature that uses Source Generators to create high-performance, structured logging code at compile time. [1, 2, 3, 4] 

Instead of parsing log strings and allocating memory during application runtime, it generates highly optimized boilerplate code before your application even runs.

------------------------------
## 1. How It Functions (The Mechanics)
When you write a normal log like _logger.LogInformation("Processing order {Id}", orderId);:

   1. .NET must parse the string "Processing order {Id}" at runtime to find the placeholders.
   2. It boxes value types (like converting an integer Id to an object) which creates garbage collection pressure.
   3. It allocates memory for the arguments array on every single log call. [5] 

When you use Source Generators (LoggerMessageAttribute):

   1. You define a partial method with an attribute.
   2. During compilation, .NET looks at that attribute and writes a strongly-typed wrapper method behind the scenes.
   3. At runtime, it skips string parsing and executes with zero memory allocations (zero garbage collection overhead). [6, 7, 8, 9] 

------------------------------
## 2. How to Implement It in Your Layers
You define these high-performance loggers as partial methods. You can group them into a single file or include them directly inside your existing Controller, Service, or Repository classes. [10, 11, 12] 

## In Your Service Layer (Service Project)

using Microsoft.Extensions.Logging;
public partial class OrderService : IOrderService
{
    private readonly ILogger<OrderService> _logger;

    public OrderService(ILogger<OrderService> logger) => _logger = logger;

    // 1. Define the Source-Generated Logging Method
    [LoggerMessage(
        EventId = 1001, // Unique ID for this log type
        Level = LogLevel.Information, 
        Message = "Processing order payment for Customer: {CustomerId} with Amount: {Amount}")]
    public partial void LogOrderProcessing(string customerId, decimal amount);

    public void ProcessOrder(OrderDto dto)
    {
        // 2. Call it like a regular typed method
        LogOrderProcessing(dto.CustomerId, dto.TotalAmount);
    }
}


* What happens behind the scenes? .NET generates a hidden file during compilation that hooks directly into the low-level ILogger architecture, bypassing string manipulations entirely.

* What about the TenantId? Because this still executes within your middleware pipeline, your TenantId and TraceId from Serilog are still automatically attached to the JSON output payload.

------------------------------
## 3. Benefits for a Shared Multi-Tenant Database Monolith

* Blazing Fast Performance: If you log frequently (e.g., tracing database table interactions for thousands of concurrent tenants), runtime log parsing will slow down your request times. Source generators eliminate this CPU and memory cost. [13, 14] 

* Strict Logging Standards: Developers can no longer make typos in log messages or forget to include critical fields. The code will fail to compile if you do not provide the correct parameters (like customerId). [15] 

* Static Analysis Support: Because every log has a unique EventId, your DevOps or infrastructure teams can easily set up alerts based on exact error codes rather than parsing unstructured text. [16] 

------------------------------
## Comparison: Standard Logging vs. Source Generators

| Feature [17, 18, 19, 20, 21] | Standard _logger.LogInformation | Source Generator (LoggerMessage) |
|---|---|---|
| String Parsing | Happens at runtime (slow) | Happens at compile time (fast) |
| Memory Allocations | Allocates memory on every call | Zero allocations for arguments |
| Type Safety | None (parameters can be mismatched) | Compile-time check (type safe) |
| TenantId Injection | Supported via middleware context | Supported via middleware context |

Yes, your thinking is 100% correct! You can design a system that forces developers to use predefined log types, ensuring consistency while automating the capture of parameters across all layers.

Here is how you can use Source Generators, Action Filters, and Interceptors to automate logging and eliminate manual guesswork for developers.
------------------------------
## 1. Forcing Predefined Parameters (The Reusable Logger Class)
To prevent developers from making mistakes or using random log formats, bundle your source-generated logs into a dedicated, reusable service. This service can be registered in your Dependency Injection (DI) container and injected wherever needed.


using Microsoft.Extensions.Logging;
public interface IAppLogger
{
    void LogControllerAccess(string controller, string action, string method);
    void LogServiceValidation(string service, string operation, string reason);
    void LogDataRowChange(string table, string operation, string primaryKey);
    void LogSystemError(string layer, string component, string exceptionMessage);
}
public partial class AppLogger : IAppLogger
{
    private readonly ILogger<AppLogger> _logger;

    public AppLogger(ILogger<AppLogger> logger) => _logger = logger;

    [LoggerMessage(EventId = 2001, Level = LogLevel.Information, Message = "[Web] Controller: {Controller} | Action: {Action} | Method: {Method}")]
    public partial void LogControllerAccess(string controller, string action, string method);

    [LoggerMessage(EventId = 3001, Level = LogLevel.Warning, Message = "[Service] {Service} | Op: {Operation} | Validation Failed: {Reason}")]
    public partial void LogServiceValidation(string service, string operation, string reason);

    [LoggerMessage(EventId = 4001, Level = LogLevel.Information, Message = "[Data] Table: {Table} | Op: {Operation} | Key: {PrimaryKey}")]
    public partial void LogDataRowChange(string table, string operation, string primaryKey);

    [LoggerMessage(EventId = 5001, Level = LogLevel.Error, Message = "[Failure] Layer: {Layer} | Component: {Component} | Error: {ExceptionMessage}")]
    public partial void LogSystemError(string layer, string component, string exceptionMessage);
}

Why this works: 
Developers simply inject IAppLogger. 
IntelliSense forces them to pass exactly the parameters required. They cannot log unstructured text, and Serilog will automatically bundle the TenantId alongside these clean parameters.

------------------------------
## 2. Automating Controller Logging (Action Filters)
You do not need to write log code in every controller action. You can use a global Action Filter in your Web project to automatically log before and after an action executes.

using Microsoft.AspNetCore.Mvc.Filters;
public class AutoLogActionFilter : IActionFilter
{
    private readonly IAppLogger _logger;

    public AutoLogActionFilter(IAppLogger logger) => _logger = logger;

    public void OnActionExecuting(ActionExecutingContext context)
    {
        var controller = context.RouteData.Values["controller"]?.ToString();
        var action = context.RouteData.Values["action"]?.ToString();
        var method = context.HttpContext.Request.Method;

        // Automatically log incoming request details
        _logger.LogControllerAccess(controller, action, method);
    }

    public void OnActionExecuted(ActionExecutedContext context)
    {
        if (context.Exception != null)
        {
            var controller = context.RouteData.Values["controller"]?.ToString();
            _logger.LogSystemError("Web", controller, context.Exception.Message);
        }
    }
}

To apply globally (in Program.cs):

builder.Services.AddControllersWithViews(options =>
{
    options.Filters.Add<AutoLogActionFilter>();
});

------------------------------
## 3. Automating Service & Repository Logging (.NET Implementations)
ASP.NET Core does not have built-in "Action Filters" for plain C# service or repository classes because filters are tied to the HTTP stack. However, you can achieve the exact same "before/after" automation using Aspect-Oriented Programming (AOP) or Interceptors.

## Option A: EF Core Interceptors (For the Repository/Data Layer)
To automate row-level changes or database query tracking without touching your Repository classes, use an EF Core SaveChanges Interceptor.

using Microsoft.EntityFrameworkCore.Diagnostics;
public class AuditLoggingInterceptor : SaveChangesInterceptor
{
    private readonly IAppLogger _logger;

    public AuditLoggingInterceptor(IAppLogger logger) => _logger = logger;

    public override ValueTask<InterceptionResult<int>> SavingChangesAsync(
        DbContextEventData eventData, InterceptionResult<int> result, CancellationToken cancellationToken = default)
    {
        var context = eventData.Context;
        if (context == null) return base.SavingChangesAsync(eventData, result, cancellationToken);

        foreach (var entry in context.ChangeTracker.Entries())
        {
            if (entry.State == EntityState.Added || entry.State == EntityState.Modified || entry.State == EntityState.Deleted)
            {
                var tableName = entry.Metadata.GetTableName();
                var operation = entry.State.ToString();
                var primaryKey = entry.Properties.FirstOrDefault(p => p.Metadata.IsPrimaryKey())?.CurrentValue?.ToString() ?? "Unknown";

                // Automatically logs every database insert/update with no code in repositories
                _logger.LogDataRowChange(tableName, operation, primaryKey);
            }
        }

        return base.SavingChangesAsync(eventData, result, cancellationToken);
    }
}

To Register (in Program.cs):

builder.Services.AddDbContext<ApplicationDbContext>((sp, options) =>
{
    options.UseSqlServer(connectionString)
           .AddInterceptors(sp.GetRequiredService<AuditLoggingInterceptor>());
});

## Option B: Dynamic Proxies / Decorators (For the Service Layer)
To automatically log before and after service methods execute without writing log code inside them, you can wrap your services in a Decorator Pattern or use a library like Castle Core (DynamicProxy).
If you want to keep it purely native .NET without external libraries, you manually create a Decorator. For example, if you have an IOrderService:

public class LoggingOrderServiceDecorator : IOrderService
{
    private readonly IOrderService _innerService;
    private readonly IAppLogger _logger;

    public LoggingOrderServiceDecorator(IOrderService innerService, IAppLogger logger)
    {
        _innerService = innerService;
        _logger = logger;
    }

    public void ProcessOrder(OrderDto dto)
    {
        // Before execution logic
        _logger.LogServiceValidation("OrderService", "ProcessOrder", $"Starting for Customer {dto.CustomerId}");

        try
        {
            _innerService.ProcessOrder(dto); // Execute real business logic
        }
        catch (Exception ex)
        {
            // Error capturing logic
            _logger.LogSystemError("Service", "OrderService", ex.Message);
            throw;
        }
    }
}

To Register in DI:

builder.Services.AddScoped<IOrderService, OrderService>();
builder.Services.Decorate<IOrderService, LoggingOrderServiceDecorator>(); 
// Uses Scrutor library, or register manually

Using an Action Filter is much better for performance, maintainability, and clean code than writing manual logging code inside every controller action. [1, 2, 3] 
There are no noticeable performance issues with running an Action Filter before and after actions, provided you avoid blocking I/O inside the filter. Here is a breakdown of why this approach works well and how it compares to manual logging. [4, 5, 6] 
------------------------------
## 1. Performance Comparison## Manual Logging Inside Actions
If you write _logger.Log... inside 50 different controller actions, you are calling the logging framework 50 separate times.

* The Problem: It doesn't actually run any faster than a filter. Furthermore, it increases your risk of memory allocations if developers accidentally use unoptimized string interpolation (e.g., $"User {id} entered" instead of structured parameters).

## Action Filters
ASP.NET Core Action Filters are highly optimized. They are instantiated via the dependency injection container and execute as a native part of the framework's MVC filter pipeline. [7, 8, 9] 

* The Cost: The execution overhead of the filter framework itself is measured in nanoseconds.
* The Catch: A performance issue only happens if you do slow, synchronous, or blocking operations inside the filter (like making an extra database query or calling an external API). As long as your filter only reads data already available in the HttpContext or RouteData and passes it to your optimized IAppLogger, it is lightning fast. [10] 

------------------------------
## 2. Why Action Filters are Better

* Guaranteed Execution: If a developer creates a new controller tomorrow, they don't have to remember to copy-paste the logging code. The filter automatically protects and logs that new code.
* Accurate Execution Timing: Action filters can easily calculate exactly how long a controller action took to run using a Stopwatch, which is tedious and messy to write manually inside every single action method. [11, 12] 
* Unified Error Capturing: If an action crashes halfway through, the manual log at the end of the action will never execute. An Action Filter's OnActionExecuted block is guaranteed to run, allowing you to capture the unhandled exception alongside the TenantId. [13, 14] 

------------------------------
## 3. The Most Performant Way to Write the Action Filter
To achieve the best possible performance, use an Asynchronous Action Filter (IAsyncActionFilter) and pass the HttpContext.Items or standard properties directly to your source-generated logger. [15] 

using System.Diagnostics;using Microsoft.AspNetCore.Mvc.Filters;
public class AutoLogActionFilter : IAsyncActionFilter
{
    private readonly IAppLogger _logger;

    public AutoLogActionFilter(IAppLogger logger) => _logger = logger;

    public async Task OnActionExecutionAsync(ActionExecutingContext context, ActionExecutionDelegate next)
    {
        // 1. Before action execution: Extract route details
        var controller = context.RouteData.Values["controller"]?.ToString() ?? "Unknown";
        var action = context.RouteData.Values["action"]?.ToString() ?? "Unknown";
        var method = context.HttpContext.Request.Method;

        _logger.LogControllerAccess(controller, action, method);

        // Start a stopwatch to track precise duration
        var stopwatch = Stopwatch.StartNew();

        // 2. Execute the actual Controller Action
        var executedContext = await next(); 

        // 3. After action execution: Stop timer and check for failures
        stopwatch.Stop();

        if (executedContext.Exception != null)
        {
            // Automatically capture crashes safely
            _logger.LogSystemError("Web", $"{controller}/{action}", executedContext.Exception.Message);
        }
        else
        {
            // Optional: You could add a log here showing execution speed if needed,
            // though Serilog's RequestLogging middleware often handles overall request timing.
        }
    }
}

## The Verdict
Do not write manual logging code inside your controllers for standard routing and entry/exit tracing. Let the Action Filter handle it globally. It keeps your controllers clean, ensures your TenantId tracking is bulletproof, and executes with near-zero performance overhead. [16, 17, 18]

Tracking HTTP request times (latency) and status codes helps you monitor your multi-tenant application's health. It answers two critical operational questions: "How fast is the app running for each customer?" and "Are certain tenants experiencing more errors than others?" [1] 
Instead of building custom timers inside Action Filters, ASP.NET Core and Serilog provide a built-in, highly optimized way to handle this globally via Serilog Request Logging Middleware.

------------------------------
## 1. The Core Metrics Explained

* HTTP Status Codes: These indicate the result of the request (e.g., 200 OK for success, 400 BadRequest for input errors, 401 Unauthorized, or 500 InternalServerError for crashes). Grouping status codes by TenantId allows you to immediately spot if a specific customer is hitting a broken feature. [2, 3, 4, 5] 
* Request Time (Latency): This is the total duration (in milliseconds) from the moment an HTTP request strikes your server until the final byte of the response is dispatched back to the user. Monitoring this helps detect performance degradation before it impacts your users. [6, 7, 8] 

------------------------------
## 2. The Built-in Solution: Serilog Request Logging
ASP.NET Core has a default logging system, but it is notoriously chatty. It writes 3 to 5 separate log lines for every single HTTP request (e.g., "Request started", "Executing action", "Executed action", "Finished request").
Serilog solves this by compressing the entire HTTP request lifecycle into one single, structured JSON log row.
## How to Add It to Your Monolith (Program.cs)
Place this line in your Program.cs file. It must be placed after your custom TenantMiddleware so that it can see the TenantId stored in the context.

var app = builder.Build();
// 1. First, resolve your tenant and push TenantId to the LogContext
app.UseMiddleware<TenantMiddleware>();
// 2. Next, inject Serilog's request logging middleware
app.UseSerilogRequestLogging(options =>
{
    // Customize the message template to fit your clean architecture style
    options.MessageTemplate = "HTTP {RequestMethod} {RequestPath} responded {StatusCode} in {Elapsed:0.0000} ms";
    
    // You can attach additional properties dynamically if needed
    options.EnrichDiagnosticContext = (diagnosticContext, httpContext) =>
    {
        diagnosticContext.Set("Host", httpContext.Request.Host.Value);
    };
});

app.UseAuthentication();
app.UseAuthorization();
app.MapControllerRoute(name: "default", pattern: "{controller=Home}/{action=Index}/{id?}");

------------------------------
## 3. What the Resulting Log Looks Like
Because this execution takes place inside your middleware pipeline envelope, Serilog automatically bundles all the contextual attributes together. When the request completes, it writes a single, clean JSON log payload to your console or log files:

  }
}

------------------------------
## 4. Why This Approach is Superior to Action Filters
Using this built-in request logging middleware is highly recommended over tracking request times via custom Action Filters for several reasons:

* Lower CPU Overhead: It hooks directly into the low-level HTTP server hosting layer, bypassing the MVC framework logic entirely for tracking.
* Catches Infrastructure Failures: If a request fails before it even reaches your controller (such as a 401 Unauthorized thrown by the Identity Cookie middleware or a 404 Not Found for a bad URL), an Action Filter will never fire. Serilog Request Logging catches every request, regardless of where it stops. [9, 10] 
* Accurate Timing: It starts measuring time the moment the request enters your application and stops only after the final response bytes are fully sent, giving you an exact metric of user-experienced delay. [11, 12] 

## Summary Strategy
For a clean .NET 8 monolith, use Serilog Request Logging to handle global HTTP speeds and success rates automatically. Then, use your type-safe Source Generators exclusively for specific business milestones inside your Service project or row changes in your Data project.

Here is exactly how those properties map to the middleware and how to ensure UserId is included alongside your TenantId.
------------------------------
## 1. How the 7 Properties are Captured

   1. Request Start & End: Handled automatically. The middleware listens for the start of the request and fires the single log line precisely when the request ends. [1] 
   2. HTTP Status: Captured automatically in the {StatusCode} property.
   3. Execution Duration: Captured automatically in the {Elapsed} property (measured in milliseconds). [2] 
   4. Unhandled Exceptions: Captured automatically. If an action crashes, the middleware catches the exception, attaches the full stack trace to the log, and automatically changes the log level from Information to Error. [3, 4, 5] 
   5. TenantId: Inherited automatically from your TenantMiddleware via the shared LogContext scope.
   6. TraceId: Captured automatically by .NET and passed along through Serilog's default tracing context.
   7. UserId: Captured by adding a small snippet inside the EnrichDiagnosticContext configuration to read the authenticated Identity Cookie claims.

------------------------------
## 2. Complete Setup for Your Monolith (Program.cs)
To make sure your UserId (from your default Cookie Identity Authentication) is extracted and passed to the log, update your UseSerilogRequestLogging configuration to look like this:

var app = builder.Build();
// 1. Resolve tenant first so it's ready in the LogContext
app.UseMiddleware<TenantMiddleware>();
// 2. Run standard Cookie Authentication so HttpContext.User is populated
app.UseAuthentication();
app.UseAuthorization();
// 3. Run Serilog Request Logging with custom enrichment
app.UseSerilogRequestLogging(options =>
{
    options.MessageTemplate = "HTTP {RequestMethod} {RequestPath} responded {StatusCode} in {Elapsed:0.0000} ms";

    // Enrich the log with properties that aren't available globally by default
    options.EnrichDiagnosticContext = (diagnosticContext, httpContext) =>
    {
        // Safely extract the UserId from the authenticated Identity Claims
        if (httpContext.User?.Identity?.IsAuthenticated == true)
        {
            var userId = httpContext.User.FindFirst(System.Security.Claims.ClaimTypes.NameIdentifier)?.Value;
            diagnosticContext.Set("UserId", userId ?? "Unknown-User");
        }
        else
        {
            diagnosticContext.Set("UserId", "Anonymous");
        }
    };
});

app.MapControllerRoute(name: "default", pattern: "{controller=Home}/{action=Index}/{id?}");

    [LoggerMessage(EventId = 3003, Level = LogLevel.Information, 
        Message = "[Service] Action: {ActionName} | Customer: {CustomerId} | External Call to {IntegrationSystem} completed with {Status}")]
    public partial void LogExternalCall(string actionName, string customerId, string integrationSystem, string status);

    [LoggerMessage(EventId = 3004, Level = LogLevel.Debug, 
        Message = "[Service] Action: {ActionName} | Customer: {CustomerId} | Feature Flag '{FlagName}' evaluates to {IsEnabled}")]
    public partial void LogFeatureFlag(string actionName, string customerId, string flagName, bool isEnabled);
}

## 3. How a Developer Uses This in a Service Class
By injecting IAppLogger into your Service classes, executing a multi-step business workflow becomes incredibly readable and strictly standardized:

public class OrderService : IOrderService
{
    private readonly IAppLogger _logger;
    private readonly IFeatureFlagService _featureFlags;

    public OrderService(IAppLogger logger, IFeatureFlagService featureFlags)
    {
        _logger = logger;
        _featureFlags = featureFlags;
    }

    public void ProcessOrder(OrderDto dto)
    {
        string action = nameof(ProcessOrder);

        // 1. Log Feature Flag Evaluation
        bool isNewPipeline = _featureFlags.IsEnabled("NewPaymentPipeline");
        _logger.LogFeatureFlag(action, dto.CustomerId, "NewPaymentPipeline", isNewPipeline);

        // 2. Log a Validation Failure
        if (dto.TotalAmount <= 0)
        {
            _logger.LogValidationFailure(action, dto.CustomerId, "Amount must be greater than zero.");
            return;
        }

        // 3. Log a Business Decision
        _logger.LogBusinessDecision(action, dto.CustomerId, "Approved routing to Stripe gateway.");

        // 4. Log an External Integration Call
        _logger.LogExternalCall(action, dto.CustomerId, "Stripe API", "Success");
    }
}

## 1. Capturing Slow Queries & Command Text
To track slow-running queries, create a DbCommandInterceptor in your Data Project. This interceptor runs automatically every time a database query finishes, checking the duration and logging a warning if it exceeds your threshold (e.g., 200ms). [5, 6, 7] 

using System.Data.Common;using Microsoft.EntityFrameworkCore.Diagnostics;using Microsoft.Extensions.Logging;
public class SlowQueryInterceptor : DbCommandInterceptor
{
    private readonly ILogger<SlowQueryInterceptor> _logger;
    private const long SlowQueryThresholdMs = 200; // Define what "slow" means to you

    public SlowQueryInterceptor(ILogger<SlowQueryInterceptor> logger)
    {
        _logger = logger;
    }

    public override void CommandExecuted(
        DbCommand command, 
        CommandExecutedEventData eventData, 
        DbCommandResult result)
    {
        LogIfSlow(command, eventData);
    }

    public override ValueTask<DbCommandResult> CommandExecutedAsync(
        DbCommand command, 
        CommandExecutedEventData eventData, 
        DbCommandResult result, 
        CancellationToken cancellationToken = default)
    {
        LogIfSlow(command, eventData);
        return base.CommandExecutedAsync(command, eventData, result, cancellationToken);
    }

    private void LogIfSlow(DbCommand command, CommandExecutedEventData eventData)
    {
        long durationMs = eventData.Duration.Milliseconds;

        if (durationMs >= SlowQueryThresholdMs)
        {
            // Structural parameters: QueryDurationMs and CommandText
            _logger.LogWarning(
                "Slow Query Detected! Duration: {QueryDurationMs}ms | SQL: {CommandText}", 
                durationMs, 
                command.CommandText);
        }
    }
}

------------------------------
## 2. Capturing Connection Issues & Deadlocks
Connection dropouts, timeouts, and SQL Server deadlocks surface as DbUpdateException or SqlException deep inside EF Core.
Instead of wrapping every repository method in try-catch blocks, you can handle these globally by using an EF Core CoreDiagnosticInterceptor or catching them inside your Global Exception Middleware / Serilog Request Logger.
When a database connection breaks or a deadlock occurs, the exception bubbles up, and Serilog automatically logs it as an Error level event, capturing the full database failure details along with the TenantId. [8] 
------------------------------
## 3. Registering the Interceptor in Your Monolith
Register the interceptor inside your Web project's Program.cs or your Data project's dependency injection configuration:

builder.Services.AddScoped<SlowQueryInterceptor>();

builder.Services.AddDbContext<ApplicationDbContext>((serviceProvider, options) =>
{
    options.UseSqlServer(builder.Configuration.GetConnectionString("DefaultConnection"))
           // Automatically injects the interceptor into EF Core
           .AddInterceptors(serviceProvider.GetRequiredService<SlowQueryInterceptor>());
});
