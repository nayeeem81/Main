Since you are using ASP.NET Core Identity, the framework automatically manages user authentication, issues the secure identity cookie, and populates User.Identity.IsAuthenticated. You do not need to manually generate JWT tokens or write custom code to validate the cookies.

However, because you are in a multi-tenant architecture, you still need your custom code for Tenant Validation to prevent "cross-tenant tenant hopping" (e.g., User A logging into Tenant 1, then manually changing the URL or header to view Tenant 2).

Here is exactly how ASP.NET Core Identity integrates into your multi-tenant session and header workflow.

The Security Gap You Must Handle

Even though Identity handles authentication perfectly, it does not natively know about your tenants. If a user is authenticated via the Identity cookie, they are authenticated across the entire app.You must use your TenantResolverMiddleware to verify that the logged-in user actually belongs to the currently requested tenant.1. 

The Multi-Tenant Identity Login

When a user logs in, you use Identity's SignInManager, but you must save the TenantId to both the User Claims and the Session at the exact same time.

csharp

[HttpPost]
[ValidateAntiForgeryToken]
public async Task<IActionResult> Login(LoginViewModel model)
{
    var resolvedTenantId = HttpContext.Items["CurrentTenantId"]?.ToString();

    // 1. Find user and ensure they belong to the current tenant context

    var user = await _userManager.FindByNameAsync(model.Username);

    if (user == null || user.TenantId != resolvedTenantId)
    {
        ModelState.AddModelError("", "Invalid login attempt for this tenant.");
        return View(model);
    }

    // 2. Use Identity to sign in (Handles the Identity Cookie automatically)

    var result = await _signInManager.PasswordSignInAsync(user, model.Password, model.RememberMe, false);

    
    if (result.Succeeded)
    {
        // 3. Lock the tenant into the server session immediately
        HttpContext.Session.SetString("TenantId", resolvedTenantId);

        return RedirectToAction("Index", "Dashboard");
    }

    ModelState.AddModelError("", "Invalid login attempt.");
    return View(model);
}

2. The Multi-Tenant Security Check in Middleware

Because Identity runs automatically, your middleware just needs to perform a quick cross-check: Does the Tenant ID inside the Identity Cookie match the Tenant ID in your Session/Header?

csharp

public class TenantResolverMiddleware
{
    private readonly RequestDelegate _next;

    public TenantResolverMiddleware(RequestDelegate next)
    {
        _next = next;
    }

    public async Task InvokeAsync(HttpContext context)
    {
        // 1. Resolve Tenant from URL / Header
        string currentTenantId = context.Request.Headers["X-Tenant-ID"].ToString();

        if (string.IsNullOrEmpty(currentTenantId))
        {
            // Fallback to domain/subdomain parsing logic if header is missing
            currentTenantId = ParseTenantFromUrl(context.Request.Host.Host);
        }

        // 2. If the user is authenticated via ASP.NET Identity, perform cross-validation

        if (context.User.Identity?.IsAuthenticated == true)
        {
            // Extract TenantId embedded in Identity Claims (or from your ApplicationUser)

            var identityTenantId = context.User.FindFirst("TenantId")?.Value;

            var sessionTenantId = context.Session.GetString("TenantId");


            // Enforce strict matching across all three points of truth

            if (currentTenantId != identityTenantId || currentTenantId != sessionTenantId)
            {
                // Tenant mismatch! Sign them out or reject the request

                context.Response.StatusCode = StatusCodes.Status403Forbidden;
                await context.Response.WriteAsync("Unauthorized: Tenant context mismatch.");
                return;
            }
        }

        // 3. Store valid tenant context for DbContext use
        context.Items["CurrentTenantId"] = currentTenantId;

        await _next(context);
    }
}

No youu do not need to encrypt the tenant ID to store it in a cookie or validate it against the session.

A plain text TenantId (like a string name, integer, or GUID) is perfectly safe to expose to the browser, provided your security architecture follows these rules:

1. Why Encryption is Not Required

The Session is Safe: Your session data resides exclusively on your backend server. The user only possesses a session identifier cookie (e.g., .AspNetCore.Session), which is cryptographically signed and unreadable. They cannot modify the server-side value of HttpContext.Session.GetString("TenantId"). 

2. The Identity Cookie is Already Encrypted: 

Since you are using ASP.NET Core Identity, the framework automatically encrypts and signs the entire identity authentication cookie. If you bake the TenantId inside the user claims, it is completely secure from tampering.

Tenant IDs are Public Identifiers: 
The TenantId is used to routing and querying data; it is not a secret credential.

The Risk: 
ID Tampering (and How to Stop It)
The only danger of an unencrypted TenantId is ID Tampering (Spoofing). If a user changes their incoming request header X-Tenant-ID from Tenant-A to Tenant-B, they are attempting to jump tenants.

Because you are cross-validating the incoming plain-text value against your secure Server Session or Identity Claims, your middleware catches this instantly and denies the request.Secure Implementation Blueprint

Your middleware can look at the plain-text tenant ID incoming from the header or URL, but it must use the server-controlled variables as the source of absolute truth:csharp

public class TenantResolverMiddleware
{
    private readonly RequestDelegate _next;

    public TenantResolverMiddleware(RequestDelegate next)
    {
        _next = next;
    }

    public async Task InvokeAsync(HttpContext context)
    {
        // 1. Grab plain-text TenantID from incoming request layer (Header or URL)
        string incomingTenantId = context.Request.Headers["X-Tenant-ID"].ToString();

        if (string.IsNullOrEmpty(incomingTenantId))
        {
            context.Response.StatusCode = StatusCodes.Status400BadRequest;
            await context.Response.WriteAsync("Tenant context missing.");
            return;
        }

        // 2. If user is logged in, validate against server-state
        if (context.User.Identity?.IsAuthenticated == true)
        {
            var sessionTenantId = context.Session.GetString("TenantId");
            var claimTenantId = context.User.FindFirst("TenantId")?.Value;

            // CROSS-VALIDATION: Plain text must match server-managed state
            if (incomingTenantId != sessionTenantId || incomingTenantId != claimTenantId)
            {
                // Request is rejected because plain-text was altered by the client
                context.Response.StatusCode = StatusCodes.Status403Forbidden;
                await context.Response.WriteAsync("Security Violation: Tenant mismatch detected.");
                return;
            }
        }

        // 3. Keep plain-text valid TenantId in context for EF Core queries
        context.Items["CurrentTenantId"] = incomingTenantId;

        await _next(context);
    }
}
