# EXCEPTION HANDELER (CROSS CUTTIG CONCERNS)

Implementing global exception handling ensures your API remains secure, predictable, and compliant with modern standards like RFC 7807 (Problem Details for HTTP APIs) 
Here is how to design and build a robust global exception handling system.

# Core Architecture Principles
1. Fail Safely: Never expose raw stack traces, database schema details, or internal server IPs to the client.
2. Standardized Output: Return a consistent JSON payload structure for every error type.
3. Internal Logging: Log the complete, unredacted error detail internally for developer debugging.
4. Unique Error IDs: Attach a correlation ID to the response so users can reference specific logs when reporting issues.
5. Standardized JSON Structure (RFC 7807) Use this production-ready blueprint for your error responses:

json

{
  "type": "https://yourdomain.com",
  "title": "An unexpected error occurred.",
  "status": 500,
  "detail": "We encountered an issue processing your request. Please try again later.",
  "instance": "/api/v1/orders/12345",
  "traceId": "c53b2611-66bb-49e0-bc4f-4d3bf2f2541a"
}

Use code with caution.

- type: A URI reference that identifies the problem type.
- title: A short, human-readable summary of the problem type.
- status: The HTTP status code generated by the origin server.
- detail: A human-readable explanation specific to this occurrence - (sanitized).instance: A URI reference that identifies the specific occurrence of the problem.
- traceId: A unique string to look up the exact error in your centralized logging system.

# Implementation Strategy
1. Catch-All Middleware
Place a middleware or interceptor at the absolute entry point of your application pipeline. This acts as a safety net for any exception that escapes the controller or business logic.
2. Exception Classification Matrix
Map known application exceptions to specific HTTP status codes, while mapping unknown errors to a generic 500 Internal Server Error.

## Error Types
1. Aplication Specific Error: HTTP status codes (Aplication Specific Error Code)
2. Unknown Errors: 500 Internal Server Error (Unknown errors to a generic Exception Type)

## User-Facing Detail Message for (Application Specific Error):
1. "EntityNotFoundException 404 Not Found": The requested resource does not exist.
2. "ValidationException 400 Bad Request": One or more validation errors occurred.
3. "UnauthorizedAccessException 401 Unauthorized": Authentication is required to access this resource.
4. "SqlException / Timeout Exception 500 Internal Server Error": An unexpected error occurred. Please try again later.
5. "Any Unhandled Exception 500 Internal Server Error": An unexpected error occurred. Please try again later.

# The Sanitization Step

When an exception hits the handler, execute a strict transformation block:text

[Incoming Exception] ──► Log full details + Stack Trace internally
                              │
                              ▼
                 Is it a known Domain Exception?
                   ├── YES ──► Use specific HTTP code + safe domain message
                   └── NO  ──► Enforce 500 Status + Hardcoded generic message
                              │
                              ▼
                    [Output RFC 7807 JSON]

Use code with caution.

# Code Blueprint (.NET C# Example)
This implementation utilizes the native IExceptionHandler feature in modern .NET to cleanly map and sanitize exceptions.

csharp

using Microsoft.AspNetCore.Diagnostics;
using Microsoft.AspNetCore.Mvc;

public class GlobalExceptionHandler : IExceptionHandler
{
    private readonly ILogger<GlobalExceptionHandler> _logger;

    public GlobalExceptionHandler(ILogger<GlobalExceptionHandler> logger)
    {
        _logger = logger;
    }

    public async ValueTask<bool> TryHandleAsync(
        HttpContext httpContext, 
        Exception exception, 
        CancellationToken cancellationToken)
    {
        var traceId = Guid.NewGuid().ToString();

        // 1. Log the internal infrastructure quirks safely inside your firewall
        
        _logger.LogError(exception, "Unhandled exception occurred. TraceId: {TraceId}. Message: {Message}", traceId, exception.Message);

        // 2. Default to a secure, generic 500 error

        var problemDetails = new ProblemDetails
        {
            Status = StatusCodes.Status500InternalServerError,
            Title = "Internal Server Error",
            Detail = "An unexpected error occurred on our end. Please contact support with your Trace ID.",
            Instance = httpContext.Request.Path,
            Type = "https://yourdomain.com"
        };

        // 3. Map specific safe exceptions if applicable
        if (exception is KeyNotFoundException)
        {
            problemDetails.Status = StatusCodes.Status404NotFound;
            problemDetails.Title = "Resource Not Found";
            problemDetails.Detail = exception.Message; 
            // Only do this if you control the message text safely
            problemDetails.Type = "https://yourdomain.com";
        }

        // 4. Inject the Trace ID for tracking purposes
        problemDetails.Extensions.Add("traceId", traceId);

        // 5. Write the standardized JSON response
        httpContext.Response.StatusCode = problemDetails.Status.Value;
        httpContext.Response.ContentType = "application/problem+json";
        
        await httpContext.Response.WriteAsJsonAsync(problemDetails, cancellationToken);

        return true;
    }
}

# Log the internal infrastructure quirks safely inside your firewall (Explain me):

That phrase refers to keeping confidential technical details about your servers, networks, and databases hidden from the public internet while saving them securely inside your internal corporate network.

Here is exactly what those terms mean in a production environment:

## Internal Infrastructure Quirks

These are sensitive configuration details and raw system responses that a hacker could use to map out your system for an attack. They include: [1] 

1. * Database Stack Traces: Errors like SqlException: Deadlock found on index IX_Orders_UserId that reveal your database engine, table names, and index strategies.
2. * Server Frameworks: Error messages explicitly mentioning versions, such as Node.js v20.11.0 or Tomcat/10.1, which tell attackers exactly what software exploits to look for.
3. * Local File Paths: Path details like C:\Users\Administrator\source\repos\api\Controllers\ that expose your server's operating system and directory structure.
4. * Internal IP Addresses: Network routings like 10.0.1.45 or private microservice URLs. [2, 3] 

## Safely Inside Your Firewall

This means sending these highly sensitive details exclusively to your secure, private logging infrastructure. [4] 


[Public Internet] ──❌── (Public User sees: "An unexpected error occurred.")
                           ▲
     FIREWALL WALL         │
───────────────────────────┼─────────────────────────────────────────────
     INTERNAL NETWORK      │ 
                           ▼
[Your Web API] ─────────► [Internal Log Server] (Records raw stack trace, IPs, & code line numbers)


Only authenticated developers, engineers, and automated monitoring tools (like Datadog, Splunk, or the ELK stack) can cross the firewall to read these logs. [5, 6] 


## Why This Separation Matters

   1. Security (Defense in Depth): By masking these quirks, you prevent "Information Disclosure" vulnerabilities. Attackers get no clues about how your code or network is structured. [7, 8] 
   2. User Experience: Your users receive clean, readable error messages instead of intimidating walls of code text.
   3. Actionable Debugging: Your engineering team still gets 100% of the raw, detailed error reports they need to fix bugs by matching the user's traceId to the internal logs.

Yes, you can absolutely separate these responsibilities into distinct middlewares, and Serilog is designed exactly for this type of filtering. [1] 

You should not mix tenant resolution, exception handling, and logging into a single middleware. Keeping them separate makes your pipeline clean, secure, and easy to maintain. [2, 3] 

Here is how to structure your .NET 8 pipeline and configure Serilog to achieve this safely.

------------------------------
## The Recommended Middleware Pipeline Order

In your Program.cs, the order of your middleware component execution matters. It must be structured like a funnel: [4] 

var app = WebApplication.CreateBuilder(args).Build();

// 1. GLOBAL EXCEPTION HANDLER (The absolute outer layer)

app.UseExceptionHandler(); // Catches EVERYTHING below it, formats JSON RFC 7807, and blocks leaks

// 2. TENANT RESOLUTION MIDDLEWARE
app.UseTenantResolution(); 
// Resolves who the tenant is and sets up the Tenant Context


// 3. SERILOG ENRICHMENT MIDDLEWARE
app.UseSerilogRequestLogging(options => {

    // Custom enrichment to attach Tenant ID to all logs automatically

});


// 4. STANDARD ROUTING & SECURITY

app.UseRouting();
app.UseAuthorization();
app.MapControllers();

------------------------------
## How Serilog Filters Errors Automatically (Keeping Quirks Inside)

Serilog separates Log Generation from HTTP Responses.
When an exception happens in your application:

   1. Serilog intercepts the raw exception object and writes the full stack trace, database names, and line numbers to your internal log destination (like a file, database, or Datadog). This happens entirely inside your firewall.
   
   2. The Exception Handler Middleware swallows the dangerous exception, creates a generic RFC 7807 ProblemDetails object, and writes only that clean JSON to the HTTP response stream back to the public internet. [5, 6, 7] 

Serilog never pushes logs to the HTTP response window unless you explicitly write code to do so. It only pushes logs to the sinks (files, consoles, cloud providers) configured in your network backend. [8, 9, 10] 

------------------------------
## Step-by-Step Implementation

## 1. The Secure Exception Handler (.NET 8 Style)

Register this handler to format the public error message and attach a TraceId. It safely logs the raw error internally using Serilog before cleansing the output.


using Microsoft.AspNetCore.Diagnostics;using Microsoft.AspNetCore.Mvc;
public class GlobalExceptionHandler : IExceptionHandler
{
    private readonly ILogger<GlobalExceptionHandler> _logger;

    public GlobalExceptionHandler(ILogger<GlobalExceptionHandler> logger)
    {
        _logger = logger;
    }

    public async ValueTask<bool> TryHandleAsync(
        HttpContext httpContext, 
        Exception exception, 
        CancellationToken cancellationToken)
    {
        var traceId = Activity.Current?.Id ?? httpContext.TraceIdentifier;

        // Serilog grabs this raw exception here and saves the full stack trace internally.

        // It stays safe inside your firewall.

        _logger.LogError(exception, "An unhandled exception occurred. TraceId: {TraceId}", traceId);

        // Sanitize the response for the public internet
        var problemDetails = new ProblemDetails
        {
            Status = StatusCodes.Status500InternalServerError,
            Title = "Internal Server Error",
            Detail = "An unexpected error occurred. Please contact support and reference the Trace ID.",
            Instance = httpContext.Request.Path
        };

        problemDetails.Extensions.Add("traceId", traceId);

        httpContext.Response.StatusCode = StatusCodes.Status500InternalServerError;
        httpContext.Response.ContentType = "application/problem+json";

        await httpContext.Response.WriteAsJsonAsync(problemDetails, cancellationToken);

        return true; 
        // Tells .NET the exception is handled; stop propagating
    }
}

## 2. Separating Serilog Request Logging (With Tenant Context)

To log HTTP requests separately from your tenant resolution logic, inject Serilog's standard middleware right after your tenant middleware. This allows Serilog to read the resolved tenant ID and stamp it on every internal log entry automatically.

// In Program.cs:

app.UseTenantResolution(); 

// First, figure out who the tenant is

// Next, fire Serilog Request logging and enrich it with the resolved Tenant Data

app.UseSerilogRequestLogging(options =>
{
    options.EnrichWithHttpRequest = (diagnosticContext, httpContext) =>
    {
        // Pull the Tenant ID from wherever your Tenant Middleware saved it (e.g., HttpContext.Items or a scoped service)

        var tenantId = httpContext.Items["TenantId"]?.ToString() ?? "Unknown";
        

        // This attaches "TenantId" as a searchable field in your internal logs!

        diagnosticContext.Set("TenantId", tenantId); 
    };
});


## 3. Registering Services in Program.cs

Make sure your dependencies are tied down correctly during application startup:

var builder = WebApplication.CreateBuilder(args);

// Use Serilog as the primary logging provider

builder.Host.UseSerilog((context, services, configuration) => configuration
    .ReadFrom.Configuration(context.Configuration)
    .ReadFrom.Services(services));


// Register your .NET 8 Exception Handler

builder.Services.AddExceptionHandler<GlobalExceptionHandler>();

builder.Services.AddProblemDetails(); 
// Required for RFC 7807 support

# Here is exactly how to solve both challenges: 

Safely propagating the Tenant ID across your middlewares, and using Serilog to dynamically separate your log files so Tenant A's internal stack traces never mix with Tenant B's files.

------------------------------

## Part 1: Propagating the Tenant ID Safely
To pass the resolved Tenant ID from your custom tenant middleware down to Serilog and the global exception handler, use a Scoped Service (often called a TenantContext). [1] 

Do not rely on static variables, as they are not thread-safe and will leak tenant data between concurrent web requests.

## 1. Define the Scoped Context

public class TenantContext
{
    public string? TenantId { get; set; }
    public string? ConnectionString { get; set; }
}

Register this in Program.cs: builder.Services.AddScoped<TenantContext>();

## 2. The Tenant Resolution Middleware

This middleware runs early in the pipeline, extracts the tenant (via header, subdomain, or query), and saves it to the scoped context. [2, 3, 4, 5] 

public class TenantResolutionMiddleware
{
    private readonly RequestDelegate _next;

    public TenantResolutionMiddleware(RequestDelegate next)
    {
        _next = next;
    }

    public async Task InvokeAsync(HttpContext context, TenantContext tenantContext)
    {
        // Example: Extract tenant from an HTTP Header
        string? tenantId = context.Request.Headers["X-Tenant-Id"].FirstOrDefault();

        // Fallback or validation logic here
        tenantContext.TenantId = tenantId ?? "default-tenant";
        
        // Continue down the pipeline
        await _next(context);
    }
}

## 3. Injecting Tenant ID into Serilog Request Logging [6] 

In Program.cs, you can map this scoped context data directly into Serilog's diagnostic context so that every normal HTTP log entry is stamped with the current tenant.

app.UseTenantResolution(); // Must run first!

app.UseSerilogRequestLogging(options =>
{
    options.EnrichWithHttpRequest = (diagnosticContext, httpContext) =>
    {
        // Resolve the scoped TenantContext safely inside the request lifetime
        var tenantContext = httpContext.RequestServices.GetRequiredService<TenantContext>();
        
        // Push property to Serilog log context
        diagnosticContext.Set("TenantId", tenantContext.TenantId ?? "Unknown");
    };
});

------------------------------

## Part 2: Isolating Tenant Logs into Separate Files

To dynamically route logs to separate files based on the TenantId property, use the Serilog.Sinks.Map NuGet package. It acts as a router inside Serilog, spinning up isolated file sinks on the fly. [7] 

## 1. Install the Required Package
Run this command in your project terminal:

dotnet add package Serilog.Sinks.Map

## 2. Configure Serilog in Program.cs [8] 

Update your Serilog configuration initialization to intercept logs, read the TenantId property, and dispatch them to specific folder paths behind your firewall. [9] 


using Serilog;using Serilog.Sinks.Map;
var builder = WebApplication.CreateBuilder(args);

builder.Host.UseSerilog((context, services, configuration) => configuration
    .ReadFrom.Configuration(context.Configuration)

    // Enriches all logs with properties from the current log context scope

    .Enrich.FromLogContext() 
    .WriteTo.Map(
        keyPropertyName: "TenantId", 
        defaultKey: "infrastructure-core", 
        configure: (tenantId, wt) => 
        {
            // This creates a dedicated log file per tenant inside your private logs folder

            wt.File(
                path: $"C:\\AppLogs\\Tenants\\{tenantId}\\log-.txt",
                rollingInterval: RollingInterval.Day,
                outputTemplate: "{Timestamp:yyyy-MM-dd HH:mm:ss.fff zzz} [{Level:u3}] (TraceId: {traceId}) {Message:lj}{NewLine}{Exception}"
            );
        })
);

## 3. Updating the Global Exception Handler

To ensure that an unhandled database or infrastructure exception ends up in the correct tenant file with its full stack trace, push the TenantId into Serilog's LogContext right before logging the exception.

using Microsoft.AspNetCore.Diagnostics;using Serilog.Context;

public class GlobalExceptionHandler : IExceptionHandler
{
    private readonly ILogger<GlobalExceptionHandler> _logger;
    private readonly TenantContext _tenantContext; // Inject your scoped tenant context

    public GlobalExceptionHandler(ILogger<GlobalExceptionHandler> logger, TenantContext tenantContext)
    {
        _logger = logger;
        _tenantContext = tenantContext;
    }

    public async ValueTask<bool> TryHandleAsync(
        HttpContext httpContext, 
        Exception exception, 
        CancellationToken cancellationToken)
    {
        var traceId = httpContext.TraceIdentifier;
        var currentTenant = _tenantContext.TenantId ?? "infrastructure-core";

        // PushProperty forces Serilog to route this error write to the matching tenant folder path

        using (LogContext.PushProperty("TenantId", currentTenant))
        using (LogContext.PushProperty("traceId", traceId))
        {
            // This contains full internal quirks/stack traces, written safely behind your firewall

            _logger.LogError(exception, "Critical unhandled exception caught for Tenant: {TenantId}.", currentTenant);
        }

        // Clean, generic public response mapping (RFC 7807)

        httpContext.Response.StatusCode = StatusCodes.Status500InternalServerError;
        httpContext.Response.ContentType = "application/problem+json";

        await httpContext.Response.WriteAsJsonAsync(new
        {
            status = 500,
            title = "Internal Server Error",
            detail = "An unexpected error occurred while processing your request.",
            traceId = traceId
        }, cancellationToken);

        return true;
    }
}

To view and search raw Serilog text files efficiently, opening them one by one in a standard text editor gets overwhelming quickly. Because your logs are split into separate multi-tenant folders, you need tools that can aggregate, filter, and tail files in real time.

Here are the best ways to view and analyze your Serilog file logs, ranked from the easiest developer tools to enterprise-ready self-hosted log viewers.

------------------------------
## Option 1: The Best Developer Tool (Seq)

Seq is a free, lightweight log server built specifically to ingest and display structured logs from Serilog. It gives you a clean web interface to filter by TenantId, traceId, or error levels without touching the raw text files. [1, 2, 3, 4, 5] 

## How to use it:

   1. Install Seq: Run it locally or on your server using Docker:
   
   docker run --name seq -d -p 5341:80 -e ACCEPT_EULA=Y datalust/seq:latest
   
   2. Update Serilog Config: Keep your file logging, but add the Seq sink right alongside it in Program.cs:
   
   dotnet add package Serilog.Sinks.Seq
   
   configuration.WriteTo.Seq("http://localhost:5341");
   
   3. View Logs: Open http://localhost:5341 in your browser. You can click on any log entry to expand the structured properties (like TenantId), view full stack traces, or type simple filters like TenantId = 'tenant-abc'. [6, 7, 8, 9, 10] 

------------------------------

## Option 2: Live Desktop Log Viewers (LogExpert or Glogg)

If you prefer to read the files directly from the C:\AppLogs\Tenants\ directory without changing your code, use a dedicated log analysis tool instead of Notepad.

* LogExpert or glogg are free, open-source windows utilities built for this. [11] 

* Why they are better: They do not lock files (allowing your application to keep writing to them), they handle massive multi-gigabyte files instantly, and they support Real-time Tailing (scrolling automatically as new errors hit the system). [12] 

* Visual Highlighting: You can configure them to highlight lines containing [ERR] or [CRT] in bright red, making it easy to scan a tenant file for crashes.

------------------------------

## Option 3: Command Line Tailing (PowerShell / Linux Terminal)

If you are logged into your host server via remote terminal and need a quick look at what a specific tenant is experiencing right now, use built-in terminal tools.

1. * To view the end of a file and watch new errors roll in live (PowerShell):
Get-Content "C:\AppLogs\Tenants\tenant-abc\log-20260701.txt" -Wait -Tail 50

2. * To search across ALL tenant folders for a specific user traceId:
Select-String -Path "C:\AppLogs\Tenants\*\*.txt" -Pattern "00-84a123bc4de567f890123456789abcde"


------------------------------

## Option 4: The Enterprise Stack (Grafana Loki + Promtail)

If your application scales up and you want a secure, centralized internal dashboard that automatically scrapes your tenant log files and displays them in a web browser behind your firewall, use Grafana Loki.

   1. Promtail runs as a background service on your application server. It watches your C:\AppLogs\Tenants/**/* directory.
   2. It automatically parses the folder structure to extract the TenantId as a metadata label.
   3. It streams the log lines securely to Grafana, where your development team can use a web UI to filter logs cleanly by environment, level, or tenant without ever logging into the actual host server. [13, 14]
