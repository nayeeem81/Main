To enable lazy loading in your Entity Framework Core setup within Program.cs, chain the UseLazyLoadingProxies() method onto your AddDbContext configuration.

Here is how to set it up step-by-step.1. 

Install the Required 

NuGet PackageBefore modifying your code, install the required proxy package using your terminal:

dotnet add package Microsoft.EntityFrameworkCore.Proxies

2. Update Program.cs
Add .UseLazyLoadingProxies() before configuring your database provider (like SQL Server, PostgreSQL, or SQLite).

using Microsoft.EntityFrameworkCore;

var builder = WebApplication.CreateBuilder(args);

// Retrieve your connection string

var connectionString = builder.Configuration.GetConnectionString("DefaultConnection");

// Register DbContext with Lazy Loading Proxies enabled

builder.Services.AddDbContext<ApplicationDbContext>(options =>
    options
        .UseLazyLoadingProxies() // <-- Adds lazy loading support
        .UseSqlServer(connectionString)); // Or UseNpgsql / UseSqlite

var app = builder.Build();
// ... remaining app configuration

3. Update Your Entity Models
For lazy loading to work correctly, EF Core must be able to inherit your classes and override their properties. You must follow these two rules for your entities:Ensure your entity classes are public and not marked as final or sealed.Mark your navigation properties as virtual.

public class Blog
{
    public int Id { get; set; }
    public string Title { get; set; }

    // Must be virtual for lazy loading to intercept it
    public virtual ICollection<Post> Posts { get; set; } 
}

public class Post
{
    public int Id { get; set; }
    public string Content { get; set; }
    public int BlogId { get; set; }

    // Must be virtual for lazy loading to intercept it
    public virtual Blog Blog { get; set; } 
}

⚠️ Crucial Performance Warning

Lazy loading can trigger the N+1 query problem. 

If you retrieve 50 Blog records and loop through them to display their Posts, EF Core will execute 51 separate database queries (1 for the blogs, and 50 individual queries for each blog's posts). If you plan to serialize these objects into JSON for a Web API, standard serializers will automatically touch every property, triggering unintentional database requests for the entire object graph.
