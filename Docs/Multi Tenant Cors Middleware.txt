## Traditional MVC Application (Server-Side)

If you created your project with standard Individual Accounts, it uses Cookie Authentication by default. No raw string tokens are passed back and forth; instead, the user's identity is encrypted and stored in a browser cookie.

### Default Identity
** Default Identity: ** ClaimsIdentity wrapped inside a ClaimsPrincipal 
** Default Storage: ** An encrypted HTTP-only cookie named .AspNetCore.Identity.Application.
** Default Lifespan: ** Sliding expiration (renews as long as the user is active) or hard expiration based on your configuration.

### Account Management Tokens (Email, Password Reset)
When ASP.NET Core Identity needs to generate a token for user verification (e.g., verifying an email or resetting a password), it uses a specialized internal token provider.

** Default Provider: ** DataProtectorTokenProvider<TUser>
** Default Lifespan: ** 1 day (24 hours).

In ASP.NET Core MVC (including .NET 8.0), CORS (Cross-Origin Resource Sharing) and Antiforgery Tokens (CSRF protection) serve completely different security purposes and must be configured together carefully when handles cross-origin requests.

### Antiforgery Tokens vs. CORS

1. CORS allows or restricts browser clients on Domain A from reading data from Domain B.
2. Antiforgery prevents a malicious website from executing state-changing actions (like POST requests) on your website using a victim's active session cookie.

### Crucial Rule: 
If your MVC View needs to make a cross-origin AJAX POST request to another origin, the standard cookie-based Antiforgery token will fail unless you explicitly share tokens via headers and configure CORS to allow credentials.

** 
By default, the Razor engine automatically injects a hidden antiforgery token into any standard HTML <form> that uses the POST method. 
**

<!-- The Form Tag Helper automatically injects the hidden token -->
<form asp-action="SubmitData" method="post">
    <input type="text" name="Username" />
    <button type="submit">Submit</button>
</form>

** 
If you are using AJAX / JavaScript, you must manually fetch the token from the view and append it to your request headers:
**

<!-- Generate token inside a hidden container or read it via C# -->
<script>
    const token = '@Antiforgery.GetTokens(HttpContext).RequestToken';

    fetch('/Home/SubmitData', {
        method: 'POST',
        headers: {
            'Content-Type': 'application/json',
            'RequestVerificationToken': token // Pass token in header
        },
        body: JSON.stringify({ username: "John" })
    });
</script>


### The Server Attributes ([ValidateAntiForgeryToken])

** 
To enforce token validation on your controllers, you apply specific filter attributes.
**

** [ValidateAntiForgeryToken]: ** 
Placed directly over a specific action method or controller. It forces the server to look for and validate the __RequestVerificationToken from either the form data or request headers.

** [AutoValidateAntiforgeryToken]: ** 
Typically applied globally in Program.cs. It automatically validates tokens on all unsafe HTTP methods (POST, PUT, DELETE, PATCH) while bypassing safe methods (GET, HEAD)

### Configuring CORS for Antiforgery (.NET 8)
If your MVC backend is accepting cross-origin requests that use cookies and antiforgery tokens, you cannot use wildcard AllowAnyOrigin() configurations. You must explicitly specify the origin and allow credentials.

**
Step A: Configure Program.cs
**

var builder = WebApplication.CreateBuilder(args);

// 1. Add CORS Policy
builder.Services.AddCors(options =>
{
    options.AddPolicy("AllowSpecificOrigin", policy =>
    {
        policy.WithOrigins("https://your-trusted-frontend.com") // Must be explicit
              .AllowAnyMethod()
              .AllowAnyHeader()
              .AllowCredentials(); // Required for cross-origin cookies/tokens
    });
});

// 2. Add Antiforgery Header Configuration
builder.Services.AddAntiforgery(options => 
{
    options.HeaderName = "X-XSRF-TOKEN"; // Expected AJAX header name
});

builder.Services.AddControllersWithViews();
var app = builder.Build();

app.UseCors("AllowSpecificOrigin"); // Must be placed BEFORE Authorization
app.UseAuthorization();

**
Step B: Secure your Controller Action
**

[ApiController]
[Route("[controller]")]
public class DataController : ControllerBase
{
    [HttpPost("submit")]
    [ValidateAntiForgeryToken] // Ensures the token sent via CORS header is valid
    public IActionResult SubmitData([FromBody] MyModel model)
    {
        return Ok(new { message = "Data saved securely!" });
    }
}

## Multi Tenant Architecture

When moving to a multi-tenant architecture in an ASP.NET Core MVC application (.NET 8.0), your configuration must change dramatically. Instead of a hardcoded, single-domain policy, you must dynamically resolve configurations based on the current tenant at runtime. Here is how you handle CORS and Antiforgery validation across multiple tenants.

** 
1. Dynamic CORS Configuration (Per-Tenant Origins) 
**
If Tenant A uses tenant-a.com and Tenant B uses tenant-b.com, you cannot use static origins. You must use a dynamic strategy using the SetIsOriginAllowed option combined with a tenant resolution service.

builder.Services.AddCors(options =>
{
    options.AddPolicy("MultiTenantCorsPolicy", policy =>
    {
        policy.AllowAnyMethod()
              .AllowAnyHeader()
              .AllowCredentials() // Required for cookie/antiforgery sharing
              .SetIsOriginAllowed(origin => 
              {
                  // 1. Inject or resolve your Tenant Service using HttpContext
                  var httpContextAccessor = builder.Services.BuildServiceProvider().GetRequiredService<IHttpContextAccessor>();
                  var httpContext = httpContextAccessor.HttpContext;
                  
                  if (httpContext == null) return false;

                  // 2. Extract tenant from host, header, or query string
                  var currentTenant = ResolveTenantFromContext(httpContext); 

                  // 3. Look up allowed domains for this specific tenant from cache/DB
                  List<string> allowedTenantDomains = GetAllowedDomainsForTenant(currentTenant);

                  return allowedTenantDomains.Contains(origin);
              });
    });
});


** 
2. Antiforgery Configuration and Cookie Isolation 
**

By default, ASP.NET Core Antiforgery uses a single global cookie name (.AspNetCore.Antiforgery.XXXXX) across the entire app infrastructure. 

In a multi-tenant environment, you have two routing scenarios that completely dictate your isolation strategies:

** 
1. Scenario A: Domain/Subdomain Isolation (://myapp.com, tenant2.com)
**

If tenants are separated by separate hosts/domains, the browser automatically isolates the cookies based on the URL context.

** Validation Strategy: ** 
Standard validation works seamlessly out of the box because the browser ensures tenant1.com cannot see or pass cookies belonging to tenant2.com.

** Important Constraint: ** 
Your Data Protection keys must be centralized (e.g., stored in Redis or Azure Key Vault) so that if your app scales horizontally, all server instances can decrypt and validate the tokens identically.

** 
2. Scenario B: Path Isolation (://myapp.com, ://myapp.com) 
**

If your tenants share the exact same domain but are separated by base paths, a major security issue occurs. By default, a cookie set on ://myapp.com can be modified or read by Javascript running on ://myapp.com. To prevent cross-tenant cookie bleeding, you must dynamically scope the Antiforgery cookie path based on the current tenant:

**
// Use IConfigureNamedOptions to dynamically set the Cookie Path at request time
builder.Services.ConfigureOptions<ConfigureAntiforgeryOptions>();
**

public class ConfigureAntiforgeryOptions : IConfigureNamedOptions<AntiforgeryOptions>
{
    private readonly IHttpContextAccessor _httpContextAccessor;

    public ConfigureAntiforgeryOptions(IHttpContextAccessor httpContextAccessor)
    {
        _httpContextAccessor = httpContextAccessor;
    }

    public void Configure(string? name, AntiforgeryOptions options)
    {
        var context = _httpContextAccessor.HttpContext;
        if (context != null)
        {
            // Extract the tenant name from the URL path (e.g., "tenant1")
            string tenantName = ExtractTenantFromPath(context.Request.Path);

            // Scope the antiforgery cookie ONLY to this tenant's path
            options.Cookie.Path = $"/{tenantName}";
            options.Cookie.Name = $".Antiforgery.{tenantName}";
        }
    }

    public void Configure(AntiforgeryOptions options) => Configure(Options.DefaultName, options);
    
    private string ExtractTenantFromPath(PathString path) { /* Parsing logic */ return "tenant1"; }
}

** 
3. Tenant Validation in Controllers 
**

Even if your CORS and Antiforgery tokens are structurally valid, a compromised token shouldn't grant cross-tenant access. You must explicitly ensure that the validated user actually belongs to the active tenant space being posted to.

**
Create a custom Action Filter to automatically double-check this for all unsafe methods:
**

public class TenantDataValidationFilter : IAsyncActionFilter
{
    private readonly ITenantService _tenantService;

    public TenantDataValidationFilter(ITenantService tenantService)
    {
        _tenantService = tenantService;
    }

    public async Task OnActionExecutionAsync(ActionExecutingContext context, ActionExecutionDelegate next)
    {
        // 1. Identify context tenant
        var contextTenantId = _tenantService.GetCurrentTenantId();

        // 2. Identify the logged-in user's claim tenant 
        var userTenantId = context.HttpContext.User.FindFirst("TenantId")?.Value;

        // 3. Throw block if there is a cross-tenant attempt
        if (userTenantId != contextTenantId)
        {
            context.Result = new BadRequestObjectResult("Cross-tenant request detected and blocked.");
            return;
        }

        await next();
    }
}

### SetIsOriginAllowed
Here is how you implement SetIsOriginAllowed dynamically in your Program.cs file. This pattern injects your database, cache, or tenant configuration service to validate origins on the fly instead of relying on a hardcoded list.

**
Add this directly to your Program.cs before builder.Build():
**

using Microsoft.AspNetCore.Cors.Infrastructure;

var builder = WebApplication.CreateBuilder(args);

// Add standard services
builder.Services.AddHttpContextAccessor();
builder.Services.AddControllersWithViews();

// Configure Dynamic Multi-Tenant CORS
builder.Services.AddCors(options =>
{
    options.AddPolicy("MultiTenantCorsPolicy", policy =>
    {
        policy.AllowAnyMethod()
              .AllowAnyHeader()
              .AllowCredentials() // Vital for cross-origin Antiforgery cookies
              .SetIsOriginAllowed(origin =>
              {
                  // 1. Manually resolve your DI services since SetIsOriginAllowed executes per request
                  var serviceProvider = builder.Services.BuildServiceProvider();
                  var httpContextAccessor = serviceProvider.GetRequiredService<IHttpContextAccessor>();
                  var context = httpContextAccessor.HttpContext;

                  if (context == null) return false;

                  try
                  {
                      // 2. Extract tenant context (e.g., from header, subdomain, or path)
                      string? tenantId = context.Request.Headers["X-Tenant-ID"].FirstOrDefault() 
                                      ?? context.Request.Host.Host;

                      if (string.IsNullOrEmpty(tenantId)) return false;

                      // 3. Resolve your DB or Cache service to find valid origins for this tenant
                      // Replace 'ITenantStore' with your actual data/config lookup service
                      var tenantStore = serviceProvider.GetRequiredService<ITenantStore>();
                      var allowedOrigins = tenantStore.GetAllowedOriginsForTenant(tenantId);

                      // 4. Return true if the browser's incoming origin is authorized for this tenant
                      return allowedOrigins.Contains(origin);
                  }
                  catch
                  {
                      // Fail closed for security if any error occurs during verification
                      return false; 
                  }
              });
    });
});

var app = builder.Build();

// Enable CORS middleware (Must be placed before Authentication and Authorization)
app.UseCors("MultiTenantCorsPolicy");

app.UseAuthentication();
app.UseAuthorization();

app.MapControllerRoute(
    name: "default",
    pattern: "{controller=Home}/{action=Index}/{id?}");

app.Run();


### Critical Rules for SetIsOriginAllowedAllowCredentials()compatibility: 

When you use .AllowCredentials() to pass cookies and antiforgery tokens across domains, the standard wildcard .AllowAnyOrigin() throws a runtime server exception. SetIsOriginAllowed acts as a dynamic workaround, explicitly accepting or rejecting a single origin per request.

**
Fail Closed Strategy: 
**
Always wrap your database/lookup code in a try-catch block inside the loop and return false if an exception occurs. 

This stops unvetted requests from leaking through if your database goes down.

**
Performance warning: 
**
BuildServiceProvider() can cause unexpected singleton behavior if overused. If your lookup database logic is resource-heavy, fetch tenant configurations from an in-memory distributed cache (like Redis) inside this filter to keep request pipelines under a millisecond.
