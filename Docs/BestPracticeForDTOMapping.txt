The best practice for mapping between ViewModels and DTOs is to maintain 
a strict separation of concerns by placing them in their respective 
layers—

1. ViewModels in the presentation/web layer and 
2. DTOs in the application/service layer. 

For the actual mapping logic, industry standards lean toward using 
specialized libraries like AutoMapper or Mapster for standard cases, 
while reserving manual mapping for high-performance or highly complex 
logic.

1. Architectural Placement Layer Isolation: 

> ViewModels should only exist in the presentation layer (UI) to hold 
screen-specific state and formatting logic (like currency or date 
formatting). 

> DTOs belong in the application or service layer to act as data 
contracts between processes.

2. Contract Stability: 

Use DTOs to create a stable contract that prevents internal domain 
changes from breaking external clients. 

3. Avoid Direct Entity Exposure: 

Never expose database entities directly to the view; always map through 
DTOs or ViewModels to prevent sensitive data leaks.

#Mapping Strategies:

Automated Libraries:

AutoMapper: 
Use the AutoMapper NuGet package to automate property 
matching based on naming conventions. It is best for reducing 
boilerplate in large projects.

Mapster:
A high-performance alternative to AutoMapper that often requires less 
configuration and offers better execution speed.

Manual Mapping:

Extension Methods: 
Write static extension methods (e.g., ToViewModel() or ToDto()) 
for full control and maximum performance.

Implicit/Explicit Operators: 
Some developers prefer implementing implicit converters within the 
ViewModel class to keep mapping logic close to the data it handles.

4. Summary of Key Principles Keep ViewModels Flat: 

ViewModels should ideally be flat structures that represent what is 
exactly on the screen, even if the underlying DTOs or models are nested.Validation: 

Perform UI-level validation on the ViewModel but re-validate business 
rules after mapping to DTOs/Entities in the application layer.

5. CQRS Pattern: 
If using Command Query Responsibility Segregation, use specialized DTOs for queries (read) and separate ViewModels for 
commands (write) to further optimize performance.



Yes, you can place DTOs in a separate Class Library, but sharing them 
across all layers (Web, Service, and Repository) violates strict clean 
architecture principles. 

In a properly decoupled architecture, DTOs should stop at the Service 
layer boundary and not enter the Repository layer. Where DTOs Belong 
in the Architecture.

[ Web / UI Layer ] 
       │  ▲  (Uses ViewModels; maps to Service DTOs)
       ▼  │
[ Service / Application Layer ]  <─── (DTOs live here or in a Shared 
                                        Contracts library)
       │  ▲  (Uses Domain Entities for business logic)
       ▼  │
[ Repository / Infrastructure Layer ] 
(Uses Database Entities / EF Core Models)

The Best Practice Breakmdown Web to Service (Yes): 
The Web layer calls the Service layer using DTOs as data contracts. 
This keeps the Web layer ignorant of internal domain logic.

Service to Repository (No): The Repository layer should accept and 
return Domain Entities or database models, not DTOs. 

The Service layer is responsible for mapping incoming DTOs into Domain 
Entities before passing them down.

Why Repositories Should Not Use DTO sLeaky Abstractions: 
DTOs are designed for data transfer (often tailored to UI or 
API contracts). 

Forcing the Repository layer to understand DTOs tightly couples 
your data access layer to your presentation needs. 

Business Logic Bypass: 
Repositories persist the state of Domain Entities. 

Domain Entities contain business rules, guards, and validation. 
Passing a DTO straight to a repository usually results in bypass 
of these crucial business rules.

Persistence Ignorance: 
The repository should only care about how to save and fetch domain data,
completely independent of how that data is displayed or transferred over 
the network. Correct Layer Dependency Structure. If you decide to extract 
your contracts into a separate project, structure your solution like this:


MyApp.Shared.Contracts (Class Library): 
Contains DTOs and Request/Response models used for Web-to-Service 
communication.

MyApp.Domain (Class Library): 
Contains core business Entities and Repository interfaces (IRepository<T>).

MyApp.Infrastructure (Class Library): 
Implements the Repository interfaces using EF Core or Dapper. 

Depends only on MyApp.Domain.MyApp.Application (Class Library): 
Contains business services. 
Depends on MyApp.Domain and 
MyApp.Shared.Contracts.

MyApp.Web (API/MVC Project): 
Presentation layer. Depends on MyApp.Application and 
MyApp.Shared.Contracts.
