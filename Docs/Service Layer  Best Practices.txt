Using a service layer (or application layer) to mediate between your web application 
and infrastructure is an excellent architectural choice. It enforces the separation of 
concerns and keeps your business logic isolated from external frameworks. 

Here is a breakdown of best practices, project responsibilities, and how to handle your 
specific sign-in and static class scenario. What Belongs in the Service Project? The 
service layer is not limited to mapping Data Transfer Objects (DTOs) and entities. It 
represents the use cases of your application. 


#################################################################


# Core Responsibilities: 

1. Orchestrating Use Cases: 
Coordinating domain entities, repositories, and external services to fulfill a specific 
user action (e.g., PlaceOrder, RegisterUser).

2. Business Validation: 
Checking rules that depend on database state (e.g., "Does this email already exist?").

3. Transaction Management: 
Ensuring that a series of database operations either all succeed or all fail together. 

4. Security & Authorization: 
Verifying if the current user has permission to execute that specific business action.

5. Mapping: 
Converting database entities into DTOs to safely expose data to the web application.


################################################################


# Best Practices for the Service Layer:

1. Keep It Framework-Agnostic: 
Do not reference web-specific concepts like HttpContext, Session, or ControllerBase 
in this layer.

2. Program to Interfaces: 
Depend on abstractions (like IUserRepository or IEmailService) rather than concrete 
implementations.

3. One DTO Per Use Case: 
Avoid reusing a single DTO for every action. Create specific DTOs 
for inputs (e.g., UpdateUserCommand) and outputs (e.g., UserSummaryDto).

4. Keep Services Stateless: Services should not hold state between requests; every method 
should receive all required data via parameters.

# Handling Sign-In Data and Static Classes

You mentioned storing session data (company, userid, currency, country) in a static 
class within a Common project. The Problem with Direct Static CallsCalling a global 
static class directly from the service layer violates the separation of concerns. 

It makes your services tightly coupled, difficult to unit test, and dangerous in 
multi-threaded web environments (where static data can accidentally leak between 
different users' requests).


The Best Practice Solution:
Instead of directly accessing the static class or manually mapping entities in the web app, use 
Dependency Injection (DI) and Abstraction. 

1. Define an Interface in the Service Layer. Create an interface that represents the 
current user's context.cs

public interface IUserContext
{
    string UserId { get; }
    string Company { get; }
    string Currency { get; }
    string Country { get; }
}


2. Implement it in the Web Layer: 
Implement this interface in your web project, drawing the data from your sign-in 
session, claims, or your Common static class safely.

3. Inject it into the Service: 
Inject IUserContext into your service constructor. Your service can now use the data 
naturally without knowing how or where it is stored.

🛠️ Step 2: Implement the Interface in the Web/MVC Project

Implement the interface inside your Web MVC project. It safely reads from 
IHttpContextAccessor (for the logged-in user details), IConfiguration 
(for appsettings.json), and handles the time zone calculations dynamically.


csharpusing Application.Interfaces;

using Microsoft.AspNetCore.Http;
using Microsoft.Extensions.Configuration;
using System.Security.Claims;

namespace WebMvc.Infrastructure
{
    public class UserContext : IUserContext
    {
        private readonly IHttpContextAccessor _httpContextAccessor;
        private readonly IConfiguration _configuration;

        public UserContext(IHttpContextAccessor httpContextAccessor, IConfiguration configuration)
        {
            _httpContextAccessor = httpContextAccessor;
            _configuration = configuration;
        }

        // Helper to safely get the current user's ClaimsPrincipal
        private ClaimsPrincipal? User => _httpContextAccessor.HttpContext?.User;

        // Retrieve claims set during the Sign-In action
        public string UserId => User?.FindFirst(ClaimTypes.NameIdentifier)?.Value ?? "System";
        public string IdentityId => User?.FindFirst("IdentityId")?.Value ?? string.Empty;
        public string Company => User?.FindFirst("Company")?.Value ?? string.Empty;
        public string Currency => User?.FindFirst("Currency")?.Value ?? string.Empty;
        public string Country => User?.FindFirst("Country")?.Value ?? string.Empty;

        // Pull values directly from appsettings.json
        public string AppEnvironment => _configuration["EnvironmentSettings:Name"] ?? "Production";

        // Replaces the Common project static method.
        // It reads the user's specific country/timezone to output the correct regional clock.
        public DateTime GetLocalNow()
        {
            string timeZoneId = Country switch
            {
                "BD" => "Bangladesh Standard Time",
                "US" => "Eastern Standard Time",
                _ => "UTC" // Safe fallback
            };

            TimeZoneInfo userTimeZone = TimeZoneInfo.FindSystemTimeZoneById(timeZoneId);
            return TimeZoneInfo.ConvertTimeFromUtc(DateTime.UtcNow, userTimeZone);
        }
    }
}

To automate setting CreatedDate and ModifiedDate using Entity 
Framework Core, you can intercept the save process directly 
inside your DbContext. This ensures audit fields are updated 
across your entire application without manual mapping in your 
services or controllers.

Here is the step-by-step code implementation using .NET 8 and 
EF Core.

## 🏷️ Step 1: 

Create an Audit Interface for Entities
Define this interface in your Domain/Entity layer. Any class 
that implements it will automatically receive auditing tracking.


namespace Domain.Common
{
    public interface IAuditableEntity
    {
        string CreatedBy { get; set; }
        DateTime CreatedDate { get; set; }
        string? ModifiedBy { get; set; }
        DateTime? ModifiedDate { get; set; }
    }
}

## 📦 Step 2: 
Apply the Interface to Your Domain Entities
Implement IAuditableEntity on any database model that requires 
tracking.

using Domain.Common;
namespace Domain.Entities
{
    public class Product : IAuditableEntity
    {
        public int Id { get; set; }
        public string Name { get; set; } = string.Empty;
        public decimal Price { get; set; }

        // Audit properties implemented from the interface
        public string CreatedBy { get; set; } = string.Empty;
        public DateTime CreatedDate { get; set; }
        public string? ModifiedBy { get; set; }
        public DateTime? ModifiedDate { get; set; }
    }
}

## 🧠 Step 3: 

Override SaveChangesAsync in Your DbContext
Inject your IUserContext interface directly into your DbContext. Before saving changes to the database, EF Core will scan the change tracker for any IAuditableEntity and modify the timestamps using your local time method.

using Application.Interfaces;
using Domain.Common;
using Domain.Entities;
using Microsoft.EntityFrameworkCore;

namespace Infrastructure.Persistence
{
    public class ApplicationDbContext : DbContext
    {
        private readonly IUserContext _userContext;

        public ApplicationDbContext(
            DbContextOptions<ApplicationDbContext> options, 
            IUserContext userContext) : base(options)
        {
            _userContext = userContext;
        }

        public DbSet<Product> Products { get; set; }

        public override Task<int> SaveChangesAsync(CancellationToken cancellationToken = default)
        {
            // Get all tracked entities that implement IAuditableEntity
            var entries = ChangeTracker
                .Entries<IAuditableEntity>()
                .Where(e => e.State == EntityState.Added || e.State == EntityState.Modified);

            // Fetch current user and regional time once per execution
            var currentUserId = _userContext.UserId;
            var currentLocalTime = _userContext.GetLocalNow();

            foreach (var entityEntry in entries)
            {
                if (entityEntry.State == EntityState.Added)
                {
                    entityEntry.Entity.CreatedDate = currentLocalTime;
                    entityEntry.Entity.CreatedBy = currentUserId;
                }
                else if (entityEntry.State == EntityState.Modified)
                {
                    // Prevent EF from overwriting the original creation details during updates
                    entityEntry.Property(x => x.CreatedDate).IsModified = false;
                    entityEntry.Property(x => x.CreatedBy).IsModified = false;

                    entityEntry.Entity.ModifiedDate = currentLocalTime;
                    entityEntry.Entity.ModifiedBy = currentUserId;
                }
            }

            return base.SaveChangesAsync(cancellationToken);
        }
    }
}

## 🎯 Step 4: 
The Clean Service Layer Result
Because EF Core handles everything behind the scenes, your 
service methods remain incredibly thin and strictly focused 
on business data.

namespace Application.Services
{
    public class ProductService
    {
        private readonly ApplicationDbContext _context;

        public ProductService(ApplicationDbContext context)
        {
            _context = context;
        }

        public async Task CreateProductAsync(CreateProductDto dto)
        {
            var product = new Product
            {
                Name = dto.Name,
                Price = dto.Price
                // NO audit properties are mapped here manually anymore!
            };

            _context.Products.Add(product);
            await _context.SaveChangesAsync(); // Automatically fills CreatedBy and CreatedDate
        }

        public async Task UpdateProductPriceAsync(int productId, decimal newPrice)
        {
            var product = await _context.Products.FindAsync(productId);
            if (product != null)
            {
                product.Price = newPrice;
                await _context.SaveChangesAsync(); // Automatically fills ModifiedBy and ModifiedDate
            }
        }
    }
}
