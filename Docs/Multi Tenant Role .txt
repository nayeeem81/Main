Your approach of separating global roles from tenant-specific permissions is common and functional, but mapping them using raw IdentityRole can become messy. Below are the answers to your specific questions, designed for ASP.NET Core MVC and .NET 8.0.

1. Is this a good practice?  Yes, conceptually. However, using the standard IdentityRole for both global and tenant roles is not recommended. It mixes global system roles with dynamic tenant roles, forcing you to write complex LINQ queries every time you check if a user is an Admin in a specific tenant.

Instead, it is a much better practice to map roles as follows:

Global Roles (using IdentityRole): GlobalAdmin, User. (Every user is assigned User globally).

Tenant Permissions (using Claims or a custom TenantUserRole mapping): Admin, ContentManager, Member.

2. The Entity Mapping Schema

To achieve this cleanly, you should attach the TenantId directly to the user (or to a User-Tenant mapping table) and inject the tenant roles as Claims upon login.

A. ApplicationUser

Add a TenantId to track which tenant they belong to, or allow it to be nullable for global-only users.


public class ApplicationUser : IdentityUser
{
    // Indicates the current active tenant for this session
    public string? CurrentTenantId { get; set; }
}


B. Recommended: User-Tenant Role mapping (if a user belongs to multiple tenants)

If a user can act as an Admin in Tenant A and a Member in Tenant B, create a custom entity:

public class TenantUser
{
    public int Id { get; set; }
    public string TenantId { get; set; }
    public string UserId { get; set; }
    public ApplicationUser User { get; set; }
    public string Role { get; set; } // Admin, ContentManager, Member
}

3. The Authorize Attribute

Do not use [Authorize(Roles = "Admin")] alone, because a tenant Admin's ID will conflict with another tenant's Admin.

Instead, use Policy-Based Authorization or evaluate a custom TenantId claim in a middleware/handler.

// Example using a Global Admin check
[Authorize(Roles = "GlobalAdmin")]
public IActionResult GlobalDashboard() => View();

// Example using a Tenant Policy
[Authorize(Policy = "TenantAdminOnly")]
public IActionResult EditTenantSettings(string tenantId) => View();

To implement this policy, check the user's tenant-specific claim during registration/login.

Multiple tenants


To support multiple tenants where a single user can have different roles across different tenants (e.g., an Admin in Tenant A, but a Member in Tenant B), you must decouple tenant authorization from the standard global ASP.NET Core Identity roles.

Here is the complete implementation strategy for .NET 8.0 ASP.NET Core MVC.

1. Updated Database Schema & DbContext

Since a user can belong to multiple tenants, you need a many-to-many junction table (TenantUserRole) that links UserId, TenantId, and the specific tenant role.

// 1. Custom User Entity
public class ApplicationUser : IdentityUser
{
    // Global properties only. No single TenantId here.
    public ICollection<TenantUserRole> TenantRoles { get; set; } = new List<TenantUserRole>();
}

// 2. Junction Entity for Multi-Tenant Roles
public class TenantUserRole
{
    public string UserId { get; set; } = null!;
    public ApplicationUser User { get; set; } = null!;

    public string TenantId { get; set; } = null!;
    
    // Stores: "Admin", "ContentManager", or "Member"
    public string TenantRoleName { get; set; } = null!; 
}

// 3. ApplicationDbContext
public class ApplicationDbContext : IdentityDbContext<ApplicationUser>
{
    public ApplicationDbContext(DbContextOptions<ApplicationDbContext> options) : base(options) { }

    public DbSet<TenantUserRole> TenantUserRoles { get; set; }

    protected override void OnModelCreating(ModelBuilder builder)
    {
        base.OnModelCreating(builder);

        // Define composite primary key for the tenant-role mapping
        builder.Entity<TenantUserRole>()
            .HasKey(tur => new { tur.UserId, tur.TenantId, tur.TenantRoleName });

        builder.Entity<TenantUserRole>()
            .HasOne(tur => tur.User)
            .WithMany(u => u.TenantRoles)
            .HasForeignKey(tur => tur.UserId);
    }
}

2. Injecting Tenant Roles as Claims (The Magic Setup)

Instead of modifying the native Identity sign-in cookie structure permanently, you inject the user's specific tenant roles into their ClaimsPrincipal during login using a custom UserClaimsPrincipalFactory.

using Microsoft.AspNetCore.Identity;
using Microsoft.Extensions.Options;
using System.Security.Claims;

public class CustomClaimsPrincipalFactory : UserClaimsPrincipalFactory<ApplicationUser>
{
    private readonly ApplicationDbContext _context;

    public CustomClaimsPrincipalFactory(
        UserManager<ApplicationUser> userManager,
        IOptions optionsAccessor,
        ApplicationDbContext context) 
        : base(userManager, optionsAccessor)
    {
        _context = context;
    }

    protected override async Task GenerateClaimsAsync(ApplicationUser user)
    {
        var identity = await base.GenerateClaimsAsync(user);

        // Fetch all tenant mappings for this specific user
        var tenantRoles = _context.TenantUserRoles
            .Where(tur => tur.UserId == user.Id)
            .ToList();

        // Format claim value as: "TenantId:RoleName" (e.g., "tenant-abc:Admin")
        foreach (var tr in tenantRoles)
        {
            identity.AddClaim(new Claim("TenantRole", $"{tr.TenantId}:{tr.TenantRoleName}"));
        }

        return identity;
    }
}

Register it in your Program.cs:

builder.Services.AddIdentity()
    .AddEntityFrameworkStores<ApplicationDbContext>()
    .AddClaimsPrincipalFactory<CustomClaimsPrincipalFactory>(); 
    // <-- Register here

3. Smart Authorization using Policy & Requirement

Do not use standard [Authorize(Roles = "Admin")] because it cannot distinguish between tenants. Instead, build a custom dynamic policy requirement that automatically checks the current tenant route parameter or header.


Step A: Create the Requirement & Handler

using Microsoft.AspNetCore.Authorization;

public class TenantRoleRequirement : IAuthorizationRequirement
{
    public string AllowedRole { get; }
    public TenantRoleRequirement(string allowedRole) => AllowedRole = allowedRole;
}

public class TenantRoleHandler : AuthorizationHandler<TenantRoleRequirement>
{
    private readonly IHttpContextAccessor _httpContextAccessor;

    public TenantRoleHandler(IHttpContextAccessor httpContextAccessor)
    {
        _httpContextAccessor = httpContextAccessor;
    }

    protected override Task HandleRequirementAsync(AuthorizationHandlerContext context, TenantRoleRequirement requirement)
    {
        // 1. Bypass check entirely if user is a GlobalAdmin
        if (context.User.IsInRole("GlobalAdmin"))
        {
            context.Succeed(requirement);
            return Task.CompletedTask;
        }

        // 2. Safely extract the current Tenant ID from Route Data (e.g., /app/{tenantId}/products)
        var routeData = _httpContextAccessor.HttpContext?.GetRouteData();
        var currentTenantId = routeData?.Values["tenantId"]?.ToString();

        if (string.IsNullOrEmpty(currentTenantId))
        {
            return Task.CompletedTask; // No tenant context found in URL
        }

        // 3. Validate against the formatted "TenantId:RoleName" claim we made earlier
        var expectedClaimValue = $"{currentTenantId}:{requirement.AllowedRole}";
        
        if (context.User.HasClaim(c => c.Type == "TenantRole" && c.Value == expectedClaimValue))
        {
            context.Succeed(requirement);
        }

        return Task.CompletedTask;
    }
}


Step B: Wire it up in Program.cs

builder.Services.AddHttpContextAccessor();
builder.Services.AddTransient<IAuthorizationHandler, TenantRoleHandler>();

builder.Services.AddAuthorization(options =>
{
    // Define unique policies maps to tenant roles
    options.AddPolicy("TenantAdmin", policy => policy.Requirements.Add(new TenantRoleRequirement("Admin")));

    options.AddPolicy("TenantContentManager", policy => policy.Requirements.Add(new TenantRoleRequirement("ContentManager")));

    options.AddPolicy("TenantMember", policy => policy.Requirements.Add(new TenantRoleRequirement("Member")));
});

4. How to Use on Controllers

Now your authorization clean, semantic, and dynamically prevents cross-tenant access violations.

// Accessible only by GlobalAdmin or a user who is an 'Admin' in the specific {tenantId} url context
[Authorize(Policy = "TenantAdmin")]
[Route("app/{tenantId}/settings")]
public class TenantSettingsController : Controller
{
    public IActionResult Index(string tenantId)
    {
        return View();
    }
}

// Accessible by GlobalAdmin, Tenant Admins, and Tenant Content Managers
[Authorize(Policy = "TenantContentManager")]
[Route("app/{tenantId}/designer")]
public class CatalogController : Controller
{
    public IActionResult EditDesign(string tenantId)
    {
        return View();
    }
}

Why this approach stands out:

Performance: Tenant role claims are loaded into the auth cookie once during login. No heavy database hits to check tenant permissions on every link click.

Global Isolation: The GlobalAdmin bypasses everything instantly without needing manual tenant mappings.

Data Integrity: Users can sign into one account but retain cleanly separated, isolated privileges across multiple tenant workspaces.

Here is how to cleanly seed your GlobalAdmin user and default IdentityRole structures at application startup in .NET 8.0.1. Create a Seeding ClassCreate a dedicated static class to separate seeding logic from your main Program.cs.

using Microsoft.AspNetCore.Identity;

public static class DataSeeder
{
    public static async Task SeedDataAsync(IServiceProvider serviceProvider)
    {
        // 1. Get required services from the isolated scope
        var roleManager = serviceProvider.GetRequiredService();
        var userManager = serviceProvider.GetRequiredService<UserManager<ApplicationUser>>();

        // 2. Define global identity roles
        string[] globalRoles = ["GlobalAdmin", "User"];

        foreach (var roleName in globalRoles)
        {
            if (!await roleManager.RoleExistsAsync(roleName))
            {
                await roleManager.CreateAsync(new IdentityRole(roleName));
            }
        }

        // 3. Define and seed the single Global Admin account
        var adminEmail = "globaladmin@yourdomain.com";
        var adminUser = await userManager.FindByEmailAsync(adminEmail);

        if (adminUser == null)
        {
            var newAdmin = new ApplicationUser
            {
                UserName = adminEmail,
                Email = adminEmail,
                EmailConfirmed = true // Bypass email verification checks for initial admin
            };

            // Specify a strong default password (ideally fetched from User Secrets/Environment Variables)
            var createAdminResult = await userManager.CreateAsync(newAdmin, "SecureP@ss123!");

            if (createAdminResult.Succeeded)
            {
                // Assign the GlobalAdmin role to this user
                await userManager.AddToRoleAsync(newAdmin, "GlobalAdmin");
            }
        }
    }
}

2. Invoke Seeding in Program.cs

Call this method directly before your application starts listening for web requests (app.Run()). Use a service scope to ensure the database resources clean up properly after execution.

var app = builder.Build();

// Configure the HTTP request pipeline
if (!app.Environment.IsDevelopment())
{
    app.UseExceptionHandler("/Home/Error");
    app.UseHsts();
}

app.UseHttpsRedirection();
app.UseStaticFiles();

app.UseRouting();

app.UseAuthentication();
app.UseAuthorization();

app.MapControllerRoute(
    name: "default",
    pattern: "{controller=Home}/{action=Index}/{id?}");

// --- Add data seeding logic right here ---
using (var scope = app.Services.CreateScope())
{
    var services = scope.ServiceProvider;
    try
    {
        await DataSeeder.SeedDataAsync(services);
    }
    catch (Exception ex)
    {
        var logger = services.GetRequiredService<ILogger<Program>>();
        logger.LogError(ex, "An error occurred while seeding the application identity data.");
    }
}

app.Run();

Key Practices Used Here:

Isolation Scope: Wrapping the execution inside app.Services.CreateScope() ensures that scoped resources like the DbContext and Identity Managers do not leak memory during initialization.

Idempotent Checks: The seeder checks RoleExistsAsync and FindByEmailAsync first. This prevents crash errors or duplicate row exceptions when running migrations on subsequent server restarts.
