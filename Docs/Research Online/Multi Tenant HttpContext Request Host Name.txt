## HttpContext (request sendby browser)

### Request.Host.Value of HttpContext 
Host is exactly what the client requested in the Host HTTP header. It does not strip or add www.; it evaluates to whichever domain name the user typed into their browser (e.g., www.sample.com or sample.com).

### Key Details to Keep in Mind:
The "www" is preserved: If the user visits www.sample.com, Request.Host.Value will return www.sample.com. If they go to sample.com, it will return sample.com.

### Port Numbers are included: 
By default, if a non-standard port is used, it returns both the host and the port (e.g., sample.com:5001). No http:// prefix: _request.Host only holds the domain and port. 

**It does not include the protocol (to get that, you need to use $"{_request.Scheme}://{_request.Host}").**

If your application requires a consistent domain (e.g., forcing everything to www.sample.com or sample.com), you can check the value of _request.Host.Value and perform a 301 Redirect using the ASP.NET Core URL

To handle potential null references safely, you can use the null-conditional operator (?.) and the null-coalescing operator (??). This prevents NullReferenceException errors if _request or Host are not yet populated during certain application lifecycle stages.

### Safe Implementation Options

**Option 1:** 
Return null if request is missing 
This approach returns null safely if the request object cannot be reached.

public string? HostName => _request?.Host.Host;

**Option 2:** 
Fallback to a default stringThis approach ensures you always get a string back, even if it is just an empty placeholder.

public string HostName => _request?.Host.Host ?? string.Empty;

### Why this is important:

During early application startup, dependency injection setup, or within certain background tasks, the HTTP context (_request) may not be fully initialized. Using these safety operators guarantees your application will not crash.

**Would you like to explore how to register and use IHttpContextAccessor to safely access the current request from inside a custom class or background service?**

To access the current HTTP request safely from a custom class or service, you must use IHttpContextAccessor.

By default, the HTTP context is only easily accessible inside Controllers or Razor Pages. This service allows you to inject it anywhere.

Step 1: Register the Service
You must register the accessor in your Program.cs file so the dependency injection system can provide it.

var builder = WebApplication.CreateBuilder(args);

// Register the HttpContextAccessor
builder.Services.AddHttpContextAccessor(); 

var app = builder.Build();

Step 2: Inject and Use Safely
Inject IHttpContextAccessor into the constructor of your custom class. Use null-conditional checks to safely extract the host name.

public class MyCustomService
{
    private readonly IHttpContextAccessor _httpContextAccessor;

    public MyCustomService(IHttpContextAccessor httpContextAccessor)
    {
        _httpContextAccessor = httpContextAccessor;
    }

    // Safely reads the host name or returns an empty string if outside an HTTP context
    public string HostName => 
        _httpContextAccessor.HttpContext?.Request.Host.Host ?? string.Empty;
}

Important Lifetime Considerations (Scoped vs Singleton): IHttpContextAccessor is registered as a Singleton, but it points to the correct, unique HTTP context for the active thread.

Background Tasks: 
If you try to use this inside a background queue or thread where there is no active web browser request, HttpContext will naturally be null. The ?? string.Empty fallback safely handles this case.

To unit test a class that uses IHttpContextAccessor, you need to mock both the accessor and the internal HttpContext structure. Using a mocking framework like Moq or NSubstitute makes this straightforward.

Unit Test Example (using Moq and xUnit)Here is how to set up the mock objects so that HostName returns your expected test domain:

using Moq;
using Microsoft.AspNetCore.Http;
using Xunit;

public class MyCustomServiceTests
{
    [Fact]
    public void HostName_Returns_Expected_Domain_When_Context_Exists()
    {
        // 1. Arrange - Setup Mock Objects
        var mockAccessor = new Mock<IHttpContextAccessor>();
        var mockContext = new Mock<HttpContext>();
        var mockRequest = new Mock<HttpRequest>();

        // 2. Setup the hierarchy: Accessor -> Context -> Request -> Host
        mockRequest.Setup(r => r.Host).Returns(new HostString("://sample.com", 8080));
        mockContext.Setup(c => c.Request).Returns(mockRequest.Object);
        mockAccessor.Setup(a => a.HttpContext).Returns(mockContext.Object);

        // 3. Act - Instantiate service with the mock accessor
        var service = new MyCustomService(mockAccessor.Object);
        var result = service.HostName;

        // 4. Assert - Verify only the host name (no port) is returned
        Assert.Equal("://sample.com", result);
    }

    [Fact]
    public void HostName_Returns_EmptyString_When_HttpContext_Is_Null()
    {
        // Arrange - Setup accessor to return null (simulating a background thread)
        var mockAccessor = new Mock<IHttpContextAccessor>();
        mockAccessor.Setup(a => a.HttpContext).Returns((HttpContext)null!);

        // Act
        var service = new MyCustomService(mockAccessor.Object);
        var result = service.HostName;

        // Assert - Verify safety fallback works
        Assert.Equal(string.Empty, result);
    }
}

The _request.Host.Host property we set up earlier never contains ://. It only extracts the domain itself (like sample.com or ://sample.com).

However, if you are reading the host name from a raw string variable or a full URL that might contain a protocol (like https://), you can strip it safely.

Here are the two best ways to do this:

Option 1: The Robust Way (Using Uri)
If your string is a full URL, letting .NET parse it is the safest approach. It automatically strips http://, https://, paths, and ports.

public string GetCleanHost(string rawUrl)
{
    if (string.IsNullOrWhiteSpace(rawUrl)) return string.Empty;

    // Add a temporary protocol if it is missing, so Uri can parse it properly
    if (!rawUrl.Contains("://"))
    {
        rawUrl = "http://" + rawUrl;
    }

    var uri = new Uri(rawUrl);
    return uri.Host; // Returns only the host name (e.g., "sample.com")
}

Option 2: The Quick Way (String Manipulation)
If you just want to blindly strip :// and everything before it from a string, you can use Split.

public string StripProtocol(string rawUrl)
{
    if (string.IsNullOrWhiteSpace(rawUrl)) return string.Empty;

    // Splits at "://" and takes everything after it
    if (rawUrl.Contains("://"))
    {
        rawUrl = rawUrl.Split(new[] { "://" }, StringSplitOptions.None)[1];
    }

    // Optional: Remove any trailing paths if they exist (e.g., ://sample.com -> sample.com)
    return rawUrl.Split('/')[0]; 
}

Understanding HttpContext?.Request.Host.Host
The Request.Host property returns a HostString object containing both the domain and the port (e.g., example.com:8080). Appending the second .Host targets the underlying string property, filtering out the port entirely.

Use Case: Domain-based multi-tenant routing, setting secure cookie domains, or restricting API access to specific origins.

Important Nuance: If your application sits behind a reverse proxy (like Nginx, IIS, or Cloudflare), this might return localhost unless you configure Forwarded Headers Middleware in ASP.NET Core.

Understanding HttpContext?.Request.Path
The Request.Path property retrieves the request path relative to your application root. It always starts with a forward slash (/).

Use Case: Custom logging middleware, URL rewriting, or checking if a user is trying to access an anonymous route like /login.

Important Nuance: It does not include PathBase (the virtual directory subfolder if your app is hosted under something like ://example.com). It also completely excludes the query string (e.g., ?id=12). To get the query parameters, use Request.QueryString.URL Breakdown
